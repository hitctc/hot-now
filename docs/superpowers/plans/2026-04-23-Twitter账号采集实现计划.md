# Twitter 账号采集实现计划

> **执行约束：** 本计划只覆盖第一阶段 Twitter 账号采集。不要顺手接入 Twitter 关键词搜索、Hacker News、B 站、微博热搜或搜狗 HTML 爬虫。

**Goal:** 在 `/settings/sources` 增加独立的 Twitter 账号管理分区，并让已启用账号通过 TwitterAPI.io 拉取最新推文，最后统一写入现有 `content_items` 内容库。

**Architecture:** RSS / 微信公众号继续使用 `content_sources`。Twitter 账号使用独立 `twitter_accounts` 表保存配置和状态；采集时通过 `collectTwitterAccountIssues` 输出与现有 RSS loader 相同语义的 `LoadedIssue[]`，再并入 `runCollectionCycle`。推文指标和作者信息写入 `content_items.metadata_json`，内容页和报告继续复用现有读取链路。

**Tech Stack:** TypeScript, Fastify, Vue 3, Ant Design Vue, SQLite, Vitest

## 状态与异常边界

- `TWITTER_API_KEY` 缺失：Twitter 账号采集标记为不可用，RSS / 微信公众号照常运行。
- 单个账号失败：只更新该账号 `last_error`，其他账号继续。
- 所有 Twitter 账号失败：本轮 collection 标记 degraded，但已有 RSS 成功时仍生成报告。
- 账号被停用：下一轮不采集。
- 删除账号：删除配置，不删除历史内容。
- 重复 tweet：依赖 `externalId` 和 `canonical_url` 防止重复入库。
- 接口返回大量旧推文：collector 过滤超过 7 天的候选，内容页仍继续用现有 24 小时窗口控制 AI 新讯。

## 风险与边界

- TwitterAPI.io 的计费和速率限制不在第一版里做复杂调度，只做每账号每轮第一页。
- 第一版不做外链正文抓取，避免一条推文引入第二层网络抓取和去重复杂度。
- 第一版不把 Twitter metrics 直接改进现有评分公式，只先入 `metadata_json`。
- 第一版不做批量导入账号。若后续账号数变多，再加批量粘贴或 CSV 导入。
