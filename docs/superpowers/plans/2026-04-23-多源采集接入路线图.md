# HotNow 多源采集接入路线图

日期：2026-04-23

背景：当前 hot-now 已有 RSS source 订阅和采集链路。下一阶段要保留 RSS 稳定来源，同时引入多源主动采集，形成“去重 / 新鲜度 / 优先级 -> AI 过滤 -> 入库”的统一采集主线。

## 目标

把 hot-now 从“RSS 订阅为主”升级为：

```text
RSS 稳定订阅源
  + Twitter 账号采集
  + Twitter 关键词搜索
  + Hacker News
  + B 站 / 微博热搜
  -> 统一候选内容
  -> 去重
  -> 新鲜度过滤
  -> 来源优先级和账号权重排序
  -> AI 过滤和摘要
  -> 入库 content_items
  -> AI 新讯 / AI 热点展示
```

## 明确不做

不接入搜狗 HTML 爬虫。

原因：

- 对 hot-now 的 AI 新闻主线帮助不大。
- HTML selector 和跳转链接维护成本高。
- 搜索结果质量不可控。
- 当前工程资源应优先投入 Twitter 账号、Twitter 搜索、Hacker News、B 站和微博热搜。

也不优先接 Google / DuckDuckGo HTML 爬虫。它们可以作为远期备选，但不是当前路线图内容。

## 执行顺序

### 阶段 1：Twitter 账号采集

这是最符合当前目标的主渠道。

采集对象：

- 海外 AI 厂商和产品账号：OpenAI、Anthropic、Google DeepMind、xAI、Meta AI、Microsoft AI、NVIDIA AI 等。
- 国内厂商和产品账号：智谱、Kimi、MiniMax、阿里通义、腾讯混元、百度文心、DeepSeek 等，优先确认它们在 X/Twitter 上是否有活跃官方账号。
- 关键人物账号：Sam Altman、Elon Musk、Andrej Karpathy、Demis Hassabis 等。

产品形态：

- 后台页面可配置账号列表。
- 支持新增、编辑、启用、停用账号。
- 支持设置账号分类、优先级、展示名称和备注。
- 不把账号列表只写死在代码里。
- Twitter 账号配置与 RSS / 微信公众号来源分开管理，避免后续多类型来源全部挤在同一张来源表和同一个列表里。

技术任务：

1. 设计独立 `twitter_accounts` 配置表。
2. 设计账号配置字段：`username`、`user_id`、`displayName`、`category`、`priority`、`include_replies`、`is_enabled`、`notes`。
3. 新增 Twitter collector，调用 TwitterAPI.io 用户最新推文接口。
4. 推文转换为统一候选内容。
5. 接入现有入库链路：去重、新鲜度、优先级、AI 过滤、写入 `content_items`。
6. `/settings/sources` 增加 Twitter 账号独立管理分区。
7. 增加 `TWITTER_API_KEY` 环境变量，写入 `.env.example`、`README.md`、`AGENTS.md`。

验收标准：

- 不配置 `TWITTER_API_KEY` 时，RSS 采集正常，Twitter 来源显示可解释错误。
- 配置 key 后，可以从至少一个账号抓到最新推文并入库。
- 重复运行不会重复写入同一条推文。
- 后台启停某个账号后，下一轮采集尊重启停状态。

### 阶段 2：Twitter 关键词搜索

用途：补漏。

关键词示例：

- `AI agent`
- `GPT`
- `Claude`
- `Gemini`
- `Qwen`
- `Kimi`
- `DeepSeek`
- `open model`
- `reasoning model`

技术任务：

1. 设计 source 类型：`twitter_search`。
2. 支持后台配置关键词、语言偏好、优先级和启停状态。
3. 使用 TwitterAPI.io advanced search。
4. 默认排除 retweets 和 replies。
5. 复用或实现质量过滤：最低 likes、retweets、views、followers。
6. 强制经过 AI 相关性过滤后再入库。

验收标准：

- 关键词来源不会压过账号来源。
- 低互动、回复、转推内容默认不入库。
- AI 过滤可以解释保留或过滤原因。

### 阶段 3：Hacker News

用途：补充开发者和技术社区侧 AI 信号。

技术任务：

1. 设计 source 类型：`hackernews_search`。
2. 使用 Algolia HN API：`https://hn.algolia.com/api/v1/search`。
3. 支持配置 query 列表和时间窗口。
4. 映射 `points`、`num_comments`、`author`、`created_at`。
5. `url` 缺失时 fallback 到 `https://news.ycombinator.com/item?id={objectID}`。

验收标准：

- 不需要 API key。
- API 失败只影响 HN 来源，不影响 RSS 和 Twitter。
- HN 条目能正确进入 AI 新讯或 AI 热点。

### 阶段 4：B 站 / 微博热搜

用途：补充国内热度。

B 站技术任务：

1. 设计 source 类型：`bilibili_search`。
2. 使用 B 站公开视频搜索 API。
3. 默认按 `pubdate` 获取最新视频。
4. 清理标题中的 `<em>` 高亮标签。
5. 映射播放、点赞、评论、弹幕等指标。

微博技术任务：

1. 设计 source 类型：`weibo_trending`。
2. 使用微博热搜 public API。
3. 按 AI 相关关键词匹配热搜榜。
4. 明确它是热搜榜匹配，不是微博全文搜索。

验收标准：

- B 站和微博默认作为补充来源，不影响 Twitter 主渠道。
- 微博只进入热点判断，不默认进入 AI 新讯主阅读层。
- 采集失败有来源级 failure，不中断整轮采集。

## 统一链路设计原则

### 1. 不再把所有来源伪装成 RSS

RSS 继续走当前 `rss_url -> feedXml -> SourceAdapter`。

非 RSS 来源新增 collector 通道：

```ts
type CollectorKind =
  | "rss"
  | "twitter_account"
  | "twitter_search"
  | "hackernews_search"
  | "bilibili_search"
  | "weibo_trending";
```

### 2. 统一候选内容模型

多源采集器统一输出候选内容，再转入现有内容链路：

```ts
type CollectedCandidate = {
  title: string;
  content: string;
  url: string;
  sourceKind: string;
  sourceId?: string;
  publishedAt?: string;
  metrics?: Record<string, number>;
  author?: {
    name?: string;
    username?: string;
    followers?: number;
    verified?: boolean;
  };
};
```

### 3. 去重策略要升级

至少支持：

- 同一来源同一 `sourceId` 去重。
- 同一来源同一 canonical URL 去重。
- Twitter 原帖和外链文章同时出现时，先保留一手账号内容，再把外链作为补充信息。

后续再考虑：

- 标题相似度归并。
- 多来源报道同一事件聚合。

### 4. AI 过滤要围绕 AI 新闻目标设计

建议 AI 过滤输出：

```ts
type AiRelevanceDecision = {
  isAiRelated: boolean;
  isOfficialSignal: boolean;
  isBreakingNews: boolean;
  noveltyScore: number;
  importanceScore: number;
  confidence: number;
  reason: string;
  summary: string;
};
```

### 5. Twitter 账号优先级最高

排序权重建议：

1. 官方 AI 厂商账号。
2. 核心人物账号。
3. Twitter 关键词搜索结果。
4. Hacker News。
5. B 站。
6. 微博热搜。
7. RSS 媒体源按现有 sourcePriority 排序。

具体权重后续实现时再落到配置或数据库字段。

2. 设计独立 `twitter_accounts` 配置表。
3. 设计 `/settings/sources` 的 Twitter 账号配置交互。
4. 设计 collector 接口和推文到 `content_items` 的映射。
