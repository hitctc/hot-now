# AI时间线实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 新增一个独立内容菜单 `AI 时间线`，只展示可回溯到官方来源的 AI 官方发布事件。

**Architecture:** `AI 时间线` 不复用 `AI 新讯 / AI 热点` 的 `content_items` 新闻流，而是新增独立 `ai_timeline_events` 表和 `src/core/aiTimeline/` 模块。第一版采用代码内置官方源白名单、手动采集、独立 API 和独立 Vue 页面，避免把官方事件和普通新闻、热点、媒体报道混在一起。

**Tech Stack:** TypeScript, Fastify, SQLite, Vue 3, Ant Design Vue, Tailwind CSS, Vitest

---

## 1. 执行约束

- 本计划只覆盖第一版 `AI 时间线`。
- 第一版必须使用用户确认的事件类型：`要闻`、`模型发布`、`开发生态`、`产品应用`、`行业动态`、`官方前瞻`。
- 第一版只收官方来源，不收媒体报道、二手解读、爆料、无官方链接的传闻。
- 第一版只做手动采集，不接入默认定时采集。
- 第一版官方源先写在代码白名单里，不做后台 CRUD。
- 第一版不把时间线事件写入 `content_items`，不参与 `AI 新讯 / AI 热点` 的评分、热点归并、source filter 和反馈池。
- 第一版不做 LLM 自动判断真假，先用官方白名单和规则分类保证可控。

## 2. 事件类型定义

事件类型使用固定枚举，后端、前端、测试和文档必须保持一致：

```ts
export const aiTimelineEventTypes = [
  "要闻",
  "模型发布",
  "开发生态",
  "产品应用",
  "行业动态",
  "官方前瞻"
] as const;
```

- `要闻`：影响范围明显更大的官方发布，例如旗舰模型、新一代模型族、重大开源、重大平台能力。
- `模型发布`：新模型、新模型版本、新模型权重、新多模态能力、模型卡片或官方模型页更新。
- `开发生态`：API、SDK、CLI、Agent 工具、开发者平台、GitHub 项目、Hugging Face / ModelScope 官方组织发布。
- `产品应用`：面向用户的产品功能更新，例如 ChatGPT、Claude、Gemini、Kimi、通义、智谱等产品能力。
- `行业动态`：官方发布的套餐、价格、地区可用性、企业能力、商业合作、合规和重要服务政策变化。
- `官方前瞻`：官方预告、官方 roadmap、官方 preview，不收无官方来源的传闻。

## 3. 成功标准

- 内容菜单出现 `AI 时间线`，点击后进入 `/ai-timeline`。
- `/ai-timeline` 页面按发布时间倒序展示官方事件。
- 页面支持按事件类型筛选、按公司筛选、标题搜索。
- 每条事件必须展示官方来源链接。
- 后台有一个手动采集入口，触发后只采集官方白名单来源。
- API 可返回空态、分页、筛选条件和事件列表。
- `npm run test` 和 `npm run build` 通过。
- `README.md` 与 `AGENTS.md` 同步说明 `AI 时间线` 的边界。

## 5. 数据模型

新增表 `ai_timeline_events`：

```sql
CREATE TABLE IF NOT EXISTS ai_timeline_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  company_key TEXT NOT NULL,
  company_name TEXT NOT NULL,
  event_type TEXT NOT NULL,
  title TEXT NOT NULL,
  summary TEXT,
  official_url TEXT NOT NULL,
  source_label TEXT NOT NULL,
  source_kind TEXT NOT NULL,
  published_at TEXT NOT NULL,
  discovered_at TEXT NOT NULL,
  importance INTEGER NOT NULL DEFAULT 50,
  raw_source_json TEXT NOT NULL DEFAULT '{}',
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(official_url)
);
```

建议索引：

```sql
CREATE INDEX IF NOT EXISTS idx_ai_timeline_events_published_at
  ON ai_timeline_events(published_at DESC);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_events_company_key
  ON ai_timeline_events(company_key);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_events_event_type
  ON ai_timeline_events(event_type);
```

字段约束：

- `event_type` 只能是 6 个固定中文枚举之一。
- `official_url` 必须是官方源白名单允许的域名或 URL 前缀。
- `published_at` 必须来自官方条目的发布时间，不能用采集时间代替。
- `discovered_at` 是系统发现时间。
- `importance` 第一版只用于页面排序辅助，不做复杂评分。

## 7. 数据流

```text
用户点击手动采集
-> POST /actions/ai-timeline/collect
-> runAiTimelineCollection
-> officialAiTimelineSources
-> 拉取官方 RSS
-> aiTimelineCollector 按标题、摘要、官方 URL 规则分类
-> aiTimelineEventRepository.upsertEvents
-> /api/ai-timeline 返回列表
-> /ai-timeline 页面展示
```

主页面读取流：

```text
/ai-timeline
-> Vue Router
-> AiTimelinePage.vue
-> readAiTimelinePage()
-> GET /api/ai-timeline?type=&company=&q=&page=
-> aiTimelineEventRepository.listAiTimelineEvents()
```

## 10. 风险与处理

- 官方源 RSS 可能不可用：第一版失败时只记录 failure，不阻断页面。
- 事件分类可能不够准：第一版用规则分类，后续再考虑人工校正或 LLM 辅助，但不能影响官方来源硬约束。
- 官方源覆盖不足：第一版先用 OpenAI 和 Google AI 打通链路，后续按“确认一个官方源，加入一个”的节奏扩展。
- `官方前瞻` 容易滑向传闻：实现上只允许官方源条目进入，不收非官方传闻。
- 不复用 `content_items` 会多一套路由和表：这是必要复杂度，用来保证“官方时间线”不被普通新闻流污染。
