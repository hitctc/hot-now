# AI时间线可靠性与管理工作台 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 `AI 时间线` 从“可展示的官方候选事件列表”升级为“可信、及时、可运营的主流 AI 重要发布事件时间线”，并把候选事件管理从 `数据收集` 页拆到独立后台工作台。

**Architecture:** 继续保留 `ai_timeline_events` 作为独立事件表，不把事件写入普通 `content_items`。本轮新增官方源健康记录、事件证据表、事件合并键、独立 `/settings/ai-timeline` 系统页和对应 API；`/settings/sources` 只保留 AI 时间线健康摘要和跳转入口，避免数据收集页继续膨胀。

**Tech Stack:** TypeScript, Fastify, SQLite, Vue 3, Ant Design Vue, Tailwind CSS, Cheerio, rss-parser, Vitest

---

## 1. 已确认需求

- 主时间线要解决真实性、可靠性、及时性，不只是展示已经抓到的旧候选。
- 主时间线默认仍展示最近 `7 天` 的 `S级 / A级` 重要事件。
- 每条主时间线事件必须能回溯到官方一手来源。
- 官方前瞻可以进入时间线，但必须标注 `尚未正式发布`。
- 国内公司必须纳入，不只覆盖 OpenAI、Google、Anthropic。
- `数据收集` 页不再承载完整 AI 时间线候选管理；AI 时间线需要独立页面或大入口。
- 推荐独立系统页，不推荐大弹窗，因为候选事件、源健康、规则说明和人工修正都属于高信息密度后台工作。

## 2. 本轮目标

本轮交付后，用户应该能看到三类变化：

1. `/ai-timeline` 主时间线更可信：
   - 只展示官方证据链完整、最近 7 天、`S/A` 级、未隐藏事件。
   - 同一发布事件如果来自多个官方入口，会合并成一个事件，而不是重复展示。
   - 页面可看到“官方证据”或“来源健康提示”，知道这条事件为什么可靠。

2. `/settings/ai-timeline` 可运营：
   - 看到官方源健康状态、最近成功时间、最近官方发布时间、失败原因。
   - 看到后台候选事件池，按公司、等级、可见状态、发布时间、来源健康筛选。
   - 可以隐藏、恢复、人工确认、修改标题、修改中文摘要、修改重要级别。

3. `/settings/sources` 变轻：
   - 只展示 AI 时间线官方源摘要。
   - 提供 `进入 AI 时间线管理` 按钮。
   - 不再直接铺开完整候选事件列表。

## 3. 明确不做

- 不接媒体报道、二手解读、无官方 URL 的爆料。
- 不为了“看起来新”引入不稳定抓取源。
- 不把官方社媒、公众号、X、微博作为本轮必需依赖；这些需要独立的登录态或平台 API 策略。
- 不引入 LLM 作为可靠性判断的必需依赖；本轮继续用可解释规则和人工修正兜底。
- 不做多人审核流、操作审计、批量导入、批量删除。
- 不直接删除历史候选事件；低质量或过期事件先通过可见性和筛选规则隔离。

## 4. 关键判断

当前 AI 时间线的问题不是“源数量不够”这么简单，而是缺少三层治理：

- `源健康治理`：不知道某个官方源是否最近成功抓取、是否已经失效、是否只抓到了旧内容。
- `事件证据治理`：同一个事件可能来自 blog、docs changelog、GitHub release、Hugging Face 模型页，现在缺少证据合并。
- `后台运营治理`：候选事件太多，继续塞在 `数据收集` 页面会影响其他来源管理，也让 AI 时间线自己的排查链路不清晰。

因此本轮先补这三层，再考虑继续扩官方源。

## 6. 数据模型

### 6.1 修改 `ai_timeline_events`

新增字段：

```sql
ALTER TABLE ai_timeline_events ADD COLUMN event_key TEXT;
ALTER TABLE ai_timeline_events ADD COLUMN reliability_status TEXT NOT NULL DEFAULT 'single_source';
ALTER TABLE ai_timeline_events ADD COLUMN evidence_count INTEGER NOT NULL DEFAULT 1;
ALTER TABLE ai_timeline_events ADD COLUMN last_verified_at TEXT;
```

索引：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_ai_timeline_events_event_key
ON ai_timeline_events(event_key)
WHERE event_key IS NOT NULL;

CREATE INDEX IF NOT EXISTS idx_ai_timeline_events_reliability_recent
ON ai_timeline_events(reliability_status, published_at DESC);
```

字段语义：

- `event_key`
  - 用于合并同一发布事件。
  - 优先由 `companyKey + detectedEntities[0] + normalizedTitleAction + YYYY-MM-DD` 生成。
  - 如果无法提取实体，退回 `companyKey + normalizedTitle + YYYY-MM-DD`。
- `reliability_status`
  - `single_source`：只有一个官方证据。
  - `multi_source`：多个官方证据交叉确认。
  - `source_degraded`：事件本身可展示，但来源近期有失败。
  - `manual_verified`：后台人工确认过。
- `evidence_count`
  - 官方证据数量，来自证据表聚合。
- `last_verified_at`
  - 最近一次采集或人工确认时间。

### 6.2 新增 `ai_timeline_event_evidence`

```sql
CREATE TABLE IF NOT EXISTS ai_timeline_event_evidence (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id INTEGER NOT NULL,
  source_id TEXT NOT NULL,
  company_key TEXT NOT NULL,
  source_label TEXT NOT NULL,
  source_kind TEXT NOT NULL,
  official_url TEXT NOT NULL,
  title TEXT NOT NULL,
  summary TEXT,
  published_at TEXT NOT NULL,
  discovered_at TEXT NOT NULL,
  raw_source_json TEXT NOT NULL DEFAULT '{}',
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(event_id, source_id, official_url),
  FOREIGN KEY (event_id) REFERENCES ai_timeline_events(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_event_evidence_event_id
ON ai_timeline_event_evidence(event_id);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_event_evidence_source_id
ON ai_timeline_event_evidence(source_id);
```

### 6.3 新增 `ai_timeline_source_runs`

```sql
CREATE TABLE IF NOT EXISTS ai_timeline_source_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  source_id TEXT NOT NULL,
  company_key TEXT NOT NULL,
  company_name TEXT NOT NULL,
  source_label TEXT NOT NULL,
  source_kind TEXT NOT NULL,
  status TEXT NOT NULL,
  started_at TEXT NOT NULL,
  finished_at TEXT,
  fetched_item_count INTEGER NOT NULL DEFAULT 0,
  candidate_event_count INTEGER NOT NULL DEFAULT 0,
  important_event_count INTEGER NOT NULL DEFAULT 0,
  latest_official_published_at TEXT,
  error_message TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_source_runs_source_recent
ON ai_timeline_source_runs(source_id, started_at DESC);

CREATE INDEX IF NOT EXISTS idx_ai_timeline_source_runs_status_recent
ON ai_timeline_source_runs(status, started_at DESC);
```

状态语义：

- `success`：源成功读取并解析。
- `failed`：网络、解析、状态码或格式异常。
- `empty`：源成功读取，但没有任何可入库候选。
- `stale`：源成功读取，但最新官方发布时间超过预期窗口。

   - 最近 7 天 S/A 展示事件数。

   - 最近一次采集时间。
   - 失败源数量。
   - 过期源数量。

   - 公司。
   - 来源名称。
   - 来源类型。
   - 最近状态。
   - 最近成功时间。
   - 最新官方发布时间。
   - 本轮候选数。
   - 本轮重要事件数。
   - 失败原因。

3. `候选事件池`
   - 筛选：公司、事件类型、重要级别、可见状态、可靠性状态、最近天数、标题搜索。
   - 列表：发布时间、公司、标题、等级、发布状态、可靠性状态、证据数、可见状态。
   - 操作：隐藏、恢复展示、人工确认、编辑。

4. `事件编辑抽屉`
   - 人工标题。
   - 人工中文摘要。
   - 人工重要级别。
   - 可见状态。
   - 官方证据链接列表。
   - 保存按钮。

5. `规则说明`
   - S/A/B/C 规则。
   - 官方前瞻规则。
   - 证据链规则。
   - 哪些内容不会进入主时间线。

## 9. 事件合并规则

新增 `src/core/aiTimeline/aiTimelineEventKey.ts`。

核心规则：

```ts
export function createAiTimelineEventKey(input: {
  companyKey: string;
  title: string;
  publishedAt: string;
  detectedEntities: string[];
}): string;
```

规则：

- 标题归一化：
  - 转小写。
  - 去掉标点。
  - 把连续空白压成一个空格。
  - 去掉 `release`、`announcing`、`introducing`、`update`、`发布`、`上线` 等动作词，只保留主体。
- 日期归一化：
  - 使用 `publishedAt` 的 `YYYY-MM-DD`。
- 实体优先：
  - 有 `detectedEntities[0]` 时，key 使用实体。
  - 没有实体时，key 使用归一化标题。

示例：

```ts
createAiTimelineEventKey({
  companyKey: "openai",
  title: "Introducing GPT-5.5",
  publishedAt: "2026-04-25T10:00:00.000Z",
  detectedEntities: ["GPT-5.5"]
});
// "openai:gpt-5-5:2026-04-25"
```

## 11. 后续官方源扩展策略

本计划完成后，再进入“官方源扩展”会更稳，因为可以看到每个源是否健康。

扩源优先级：

1. 优先补官方 release notes / docs changelog。
2. 再补官方 blog / newsroom。
3. 再补 Hugging Face / GitHub 官方组织。
4. 最后才考虑官方社媒、微博、公众号。

每个新源进入生产采集前必须满足：

- 可公开访问。
- 不依赖登录态。
- 能拿到明确官方发布时间。
- 能拿到官方 URL。
- 能稳定解析至少最近 10 条。
- 失败时不会影响其他源采集。

## 12. 风险与缓解

- 风险：`event_key` 合并误伤，把两个不同事件合并。
  - 缓解：第一版只在同公司、同实体、同日期时合并；无实体时保守退回 URL 去重。

- 风险：源健康表增加后采集链路变复杂。
  - 缓解：源健康记录只写运行结果，不参与采集决策；主时间线展示仍由事件表控制。

- 风险：独立后台页面范围膨胀。
  - 缓解：第一版只做查看、筛选、单条编辑、隐藏/恢复；不做批量操作和审计。

- 风险：旧 `数据收集` 页测试大量变化。
  - 缓解：先保持手动采集动作不变，只移动候选管理 UI。

## 13. 成功标准

- `/ai-timeline` 默认展示的事件更接近“最近 7 天国内外重要 AI 发布”，不是官方源流水账。
- 每条主时间线事件有清晰官方证据数量和官方链接。
- 后台能看出哪些源过期、哪些源失败、哪些源最近没有新内容。
- 候选事件不再堆在 `数据收集` 页。
- `/settings/ai-timeline` 成为 AI 时间线唯一的运营后台入口。
