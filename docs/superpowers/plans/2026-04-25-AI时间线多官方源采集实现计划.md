# AI时间线多官方源采集实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 AI 时间线从只依赖两个 RSS feed，扩展为可维护的官方多来源采集。

**Architecture:** 保持 `ai_timeline_events` 独立事件表不变，只扩展 `src/core/aiTimeline/` 的官方源目录和采集适配器。第一批只接入无需登录、能拿到明确发布时间的一手来源，暂不接入官方社媒、公众号和需要浏览器态的站点。第二轮已补齐更多稳定官方入口，仍然把 403 / Cloudflare / 无清晰时间点的入口挡在生产采集之外。

**Tech Stack:** TypeScript, Fastify, SQLite, Vue 3, Cheerio, rss-parser, Vitest

---

## 1. 执行边界

- 仍然只收官方来源，不收媒体报道、二手转述和无官方原文的传闻。
- 每条事件必须有官方 URL 和明确发布时间；没有发布时间的条目直接跳过。
- 第一批新增来源只覆盖稳定 HTTP 可读来源：官方 RSS、官方 changelog 页面、官方博客列表、Hugging Face 官方组织模型页、GitHub releases Atom。
- OpenAI `platform.openai.com/docs/changelog` 当前静态请求会触发 Cloudflare challenge，暂不接入生产采集，避免后台长期报错。
- OpenAI `help.openai.com` 的 ChatGPT Release Notes 和 `x.ai/news` 当前服务器侧静态请求也会触发 Cloudflare challenge，暂不接入生产采集。
- GitHub REST API 当前未认证请求容易触发 rate limit；已改用 GitHub 官方 releases Atom，不依赖 REST API token。
- 设置页展示必须来自同一份官方源目录，避免前端手写清单再次过期。

## 5. 后续扩展

- 如果要接 OpenAI API changelog、ChatGPT Release Notes 或 xAI News，需要先解决 Cloudflare challenge 或找到官方可机器读取的公开版本。
- 如果要接更多 GitHub 来源，优先使用 releases Atom；只有需要 REST API 时才加可选 `GITHUB_TOKEN`。
- 腾讯 / 百度 / 小米放第二批，先确认稳定官方入口和明确发布时间，不要为了数量贸然引入噪音。
- 如果要接官方微博、X、公众号，需要单独做浏览器态或平台 API 方案，不应塞进当前 HTTP 采集器。
