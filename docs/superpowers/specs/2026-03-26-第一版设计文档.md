# HotNow 第一版设计文档

## 1. 目标与范围

### 1.1 项目目标

HotNow 第一版要跑通一条完整闭环：

`每日定时 / 手动触发 -> 拉取 RSS -> 抓取原文 -> 规则识别热点 -> 生成 HTML 报告 -> 网页查看 -> 邮件发送`

这是一款本地单机运行、面向个人使用的热点日报工具。第一版重点是把流程做稳，而不是把功能做大。

### 1.2 已确认输入

- 数据源固定为 `https://imjuya.github.io/juya-ai-daily/rss.xml`
- 第一版采用单一真实源，但内部接口按可插拔方式设计，后续允许新增数据源
- 报告按每天固定一次自动生成，同时支持手动触发
- 热点识别采用规则方式，不接入大模型
- 报告面向个人信息浏览，优先强调快速扫读和值得关注原因
- 邮件发送使用 QQ 邮箱 `SMTP + 授权码`
- 网页展示需要保留每天一份历史报告
- 每日报告输出 `Top 10` 热点主题
- 配置方式采用“配置文件 + 网页手动触发”
- 第一版复用源中已有的分类和排序，再做自己的主题归并

### 1.3 第一版明确不做

- 多数据源聚合
- LLM 聚类或生成式总结
- 登录、多用户、权限体系
- 网页直接修改配置文件
- 趋势分析、周报、月报
- 云部署、多实例、高可用
- Dashboard 级复杂前端交互
- 复杂搜索、标签、推荐系统

## 2. 数据源现状与设计约束

### 2.1 数据源结构

当前 RSS 的最新 `item` 不是单条新闻，而是一整份 AI 日报。其正文里已经包含：

- 报告日期
- 分组栏目，例如“要闻”“模型发布”“开发生态”
- 每条热点标题
- 源排序编号，例如 `#1` 到 `#20`
- 对应原始链接
- 每条热点的扩展说明和相关链接

这意味着第一版不需要先解决“大规模新闻采集”，而要先做好“从单份日报中抽取候选热点，再做二次抓取和规则归并”。

### 2.2 核心约束

- 数据源是外部公开 RSS，结构后续可能变化，所以解析逻辑必须可测试、可降级
- 原始链接来自不同站点，抓取成功率不稳定，因此必须允许单条降级
- 第一版为本地单机运行，设计应尽量减少额外基础设施依赖
- 敏感信息如 QQ 邮箱授权码不能写入仓库，必须通过环境变量注入

## 3. 系统架构

### 3.1 架构选择

第一版采用“轻量本地服务 + 文件归档”的方案：

- 一个本地单进程应用承载调度、抓取、聚类、报告生成、网页展示和邮件发送
- 报告结果以文件形式归档，不先引入数据库
- 网页承担“查看报告 + 手动触发”的最小交互职责

这个方案最符合当前场景：

- 本地单机运行
- 面向个人使用
- 需要定时任务
- 需要可追溯历史
- 需要手动触发

### 3.2 模块划分

#### 调度入口

负责定时执行和手动执行，统一进入同一套生成流水线。这个模块只负责“何时执行”和“防止重复执行”，不负责业务处理。

#### 数据源适配器

负责拉取 RSS、解析最新日报条目，并输出规范化的候选热点列表。它只关心源格式，不承担主题识别逻辑。

#### 原文抓取器

负责根据日报中的原始链接抓取正文，并对结果做清洗、长度控制和失败记录。它的职责是尽可能拿到可用文本，而不是决定热点价值。

#### 规则识别器

负责结合源排序、源分类、标题关键词、链接域名和候选条目关系，完成去重、归并、主题聚类和排序。这个模块产出最终 `Top 10` 热点主题。

#### 规则总结器

负责把结构化结果转成稳定的摘要字段，例如“发生了什么”“为什么值得关注”“原文链接”。第一版只做模板化总结，不进行开放式生成。

#### 报告生成器

负责把结果输出为 JSON 和 HTML 两份产物。JSON 供内部流程和后续扩展使用，HTML 供网页展示和邮件复用。

#### 邮件通知器

负责通过 QQ 邮箱 SMTP 发送日报邮件。它不影响主报告产出，发送失败时只记录失败状态。

#### 网页展示层

负责展示最新报告、历史报告和控制入口。它不直接修改核心配置，也不承担抓取和聚类业务。

## 4. 执行链路

### 4.1 主流程

主流程固定为：

1. 调度器或网页手动触发一次任务
2. 拉取 RSS 并定位最新日报
3. 解析日报中的栏目、排名、候选热点和原始链接
4. 对原始链接进行正文抓取和正文清洗
5. 按规则做主题归并和排序
6. 生成 `Top 10` 热点主题
7. 落结构化 JSON
8. 生成 HTML 报告
9. 更新网页可查看的最新报告与历史索引
10. 尝试发送邮件
11. 写入运行记录

### 4.2 幂等和重复执行

- 自动任务和手动任务共用同一条执行链路
- 同一天已有成功报告时，允许再次执行
- 再次执行默认覆盖该日期的最新报告产物，但保留新的运行记录
- 同一时刻只允许一个执行任务，避免手动和自动同时运行导致冲突

## 5. 热点识别与总结规则

### 5.1 输入信号

第一版明确复用源内已有信号：

- 栏目分类
- 排名编号
- 标题文本
- 原始链接

在此基础上叠加我们自己的规则：

- 标题关键词重合
- 相同主体名词，例如公司名、模型名、产品名
- 相同链接域名
- 同一主题被多个来源提及

### 5.2 主题归并

规则识别器对候选条目进行归并时，优先参考以下维度：

- 是否明显指向同一产品或模型
- 是否属于相同事件的不同报道
- 是否在同一栏目中且标题实体高度重合
- 是否存在相同或近似的核心关键词

第一版不追求复杂语义聚类，只要求“明显同主题的条目尽量聚在一起，不相关条目不要误并”。

### 5.3 排序规则

主题得分由以下信号组成：

- 源内排名越靠前，权重越高
- 高优先分类，例如“要闻”，可获得额外权重
- 同一主题下关联条目越多，权重越高
- 命中高价值关键词时可增加少量权重

最终按综合得分生成 `Top 10` 热点主题。

### 5.4 模板化总结

每个主题输出以下固定字段：

- 主题标题
- 综合排序
- 所属分类
- 发生了什么
- 为什么值得关注
- 关键词
- 关联条目数量
- 相关原文列表

总结逻辑只做结构化提取和模板拼装：

- 如果抓到原文正文，则优先从原文前部有效内容和标题中提炼
- 如果原文抓取失败，则退化为使用 RSS 中的标题和日报说明生成短总结

## 6. 报告结构与归档

### 6.1 报告结构

每份日报由三层组成：

#### 报告元信息

- 报告日期
- 生成时间
- 触发方式，自动或手动
- 数据源标识
- 抓取状态
- 热点主题数量
- 邮件发送状态
- 是否为降级报告

#### 热点主题 Top 10

每个主题包含：

- 主题标题
- 排名
- 分类
- 摘要
- 值得关注原因
- 关键词
- 关联原始条目数

#### 主题下的原始条目

每个原始条目保留：

- 源排名
- 源分类
- 原始标题
- 原文链接
- 原文站点
- 正文摘要
- 抓取是否成功

### 6.2 归档规则

第一版采用按日期归档的文件结构，每天一个目录。这个目录承载报告内容和运行结果，方便回看、补发和排错。

建议的产物形式：

- `YYYY-MM-DD/report.json`
- `YYYY-MM-DD/report.html`
- `YYYY-MM-DD/run-meta.json`

### 6.3 历史保留

- 网页端保留每天一份历史报告
- 历史列表按日期倒序展示
- 第一版不限制保留天数，后续可通过配置增加清理策略

## 7. 网页页面与交互

### 7.1 页面范围

第一版网页只做四类视图：

#### 首页 / 最新报告

- 展示最新一次生成成功的报告
- 顶部显示日期、生成时间、触发方式和邮件状态
- 主体展示 `Top 10` 热点主题
- 每个主题支持展开查看关联原始条目

#### 历史报告列表

- 按日期倒序展示历史报告
- 每项显示日期、主题数、是否发送成功、是否为降级报告

#### 单日报告详情

- 展示某一天的完整报告内容
- 展示方式与首页主体相近，但不承担“最新”语义

#### 设置/控制页

- 显示当前生效配置快照
- 提供“立即生成一次”按钮
- 可选提供“重发今日邮件”按钮

### 7.2 状态展示

网页需要清楚展示以下状态：

- 今日已有报告
- 今日尚未生成
- 当前任务执行中
- 当前任务执行失败
- 报告为降级生成
- 邮件发送失败但报告已生成

### 7.3 手动触发规则

- 网页允许触发一次即时生成任务
- 任务执行中时，禁止再次触发
- 手动触发完成后，刷新最新报告和状态信息
- 第一版不支持网页上任意日期重跑

## 8. 配置与敏感信息

### 8.1 配置方式

第一版采用“配置文件 + 环境变量”组合：

#### 配置文件

用于保存非敏感配置，例如：

- 是否启用自动调度
- 每日执行时间
- Top N 数量
- 网页端口
- 是否允许网页手动触发
- 是否允许生成降级报告

#### 环境变量

用于保存敏感信息，例如：

- SMTP 主机
- SMTP 端口
- 发件人地址
- 收件人地址
- QQ 邮箱授权码

### 8.2 网页配置边界

- 网页只展示当前生效配置，不直接编辑配置文件
- 敏感字段不在网页明文显示
- 第一版不做配置在线保存能力

## 9. 失败处理与降级策略

### 9.1 RSS 拉取失败

- 本次任务直接失败
- 不生成报告
- 写入失败运行记录
- 不发送正式日报邮件

### 9.2 RSS 可用但正文解析失败

- 尝试从 `description` 中生成一份简化报告
- 在报告元信息中标记为降级报告

### 9.3 原文抓取部分失败

- 保留对应热点条目
- 使用日报中的标题和说明做退化摘要
- 报告继续生成

### 9.4 邮件发送失败

- 不影响报告文件落地
- 网页仍可查看报告
- 在运行记录和页面状态中明确标记邮件失败

### 9.5 自动与手动触发冲突

- 同一时刻只允许一个任务执行
- 第二个触发请求应立即收到“已有任务运行中”的反馈

## 10. 测试策略

第一版不追求铺满所有测试类型，但必须覆盖行为变化最大的路径。

### 10.1 RSS 解析测试

- 能正确抽出最新日报
- 能从正文中抽出栏目、排名、标题和原始链接

### 10.2 原文抓取与清洗测试

- 能对典型文章页面提取正文
- 抓取失败时能正确返回降级结果

### 10.3 规则归并测试

- 同主题条目能归并
- 非同主题条目不会误归并
- 排序规则输出稳定可预期

### 10.4 报告生成测试

- 能生成结构化 JSON
- 能生成 HTML 报告
- 在部分原文缺失时仍能产出可用报告

### 10.5 通知测试

- 邮件内容能正确拼装
- SMTP 失败不会破坏主流程

### 10.6 基础集成测试

- 一次完整任务执行后，能产出报告文件和运行状态记录

## 11. 目录与边界建议

虽然第一版尚未进入实现，但代码组织需要提前保持可插拔。建议围绕职责拆分，而不是把流程堆进一个大文件中。

建议至少保留这些边界：

- `source adapter`
- `article fetcher`
- `topic recognizer`
- `report builder`
- `mailer`
- `scheduler`
- `web routes`

这样后续即使新增更多 RSS 源、加入 SQLite、增加新投递渠道，也不需要重写整条链路。

这份设计文档确认后，下一阶段应进入 implementation plan，至少覆盖以下内容：

- 技术栈选择

- 调度实现方式
- 页面实现顺序
- 测试落点
- 本地运行和配置说明
