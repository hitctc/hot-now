# HotNow 赛博主题改造设计文档

## 1. 目标与范围

### 1.1 当前问题

当前项目的统一站点已经具备功能闭环，但视觉呈现仍停留在“基础后台”阶段：

- 壳层结构清晰，但品牌气质和内容氛围较弱
- 明暗主题没有切换能力
- 内容卡片和系统卡片的层次接近，缺少科技感区分
- legacy 报告页虽然可用，但与统一站点的视觉语言割裂

这会直接影响产品完成度。当前需要解决的不是功能缺失，而是把项目从“能用”提升到“像一个正式的科技内容产品”。

### 1.2 新目标

本阶段目标是：

`把 HotNow 统一站点升级为具备赛博实验感、支持手动明暗切换、并且视觉上统一的科技内容控制台`

其中：

- 深色与浅色都必须是完整主题，不是简单反色
- 统一站点是视觉改造主战场
- legacy 报告页只做轻量跟随，不重做结构
- 不改变当前 Fastify SSR 架构和已有业务交互接口

### 1.3 本阶段已确认决策

- 视觉方向采用 `赛博实验感`
- 深色与浅色主题都同等强调
- 主题切换入口放在左侧导航底部
- 主题选择通过浏览器本地存储记住
- 覆盖范围为：
  - 统一站点全覆盖
  - legacy 报告页轻量跟随

### 1.4 本阶段明确不做

- 不改业务信息架构
- 不引入新的前端框架
- 不接入动画库、图表库或 canvas 特效
- 不把 SSR 页面重写成 SPA
- 不在本阶段增加新的菜单、组件系统或复杂 dashboard 数据模块

## 2. 视觉方向

### 2.1 主题定位

这次视觉定位不是“深色后台”，而是 `Cyber Intelligence Console`：

- 更像科技情报控制台，而不是普通管理系统
- 视觉重点在氛围层、模块边界、状态高亮和阅读层次
- 保持可读性，不走纯展示型炫技路线

### 2.2 深色主题

深色主题作为赛博感的主表达，视觉语言包含：

- 深墨蓝、冷黑、石墨灰作为底色
- 青蓝、酸性青绿、少量洋红作为高亮
- 带有网格、扫描感、柔和辉光和半透明分层背景
- 卡片边缘、标题区和按钮状态有更强的控制台感

深色主题应该让用户第一眼感受到：

- 这是一个科技内容工作台
- 不是通用 SaaS 后台模板

### 2.3 浅色主题

浅色主题不能只是把背景换成白色。它需要保留同样的科技气质：

- 主底色采用冷银、冰蓝和低饱和灰白
- 保留青蓝系高亮和模块边界
- 背景上仍有轻量科技纹理，例如软网格、微弱发光或实验室面板感
- 交互按钮、卡片描边和导航层次仍然保持明显个性

浅色主题的目标是：

- 看起来像实验室控制界面
- 而不是普通资讯网站

## 3. 页面改造范围

### 3.1 统一壳层

重点改造统一壳层：

- 左侧导航升级成更像“模块面板”的结构
- 品牌区从当前纯文本头变成更完整的科技品牌头
- 在左侧导航底部加入主题切换组件
- 主内容区加入背景氛围层和更强的层次结构

壳层是本次改造的视觉锚点，所有子页面都必须依赖这套壳层建立统一风格。

### 3.2 内容菜单页

内容菜单页保留现有业务结构，但视觉上要升级为“情报卡流”：

- 内容卡片边界、阴影和背景要更具科技感
- 标题、来源、发布时间和评分形成清晰的信息层级
- 收藏、点赞、点踩和评分表单进入统一主题语言
- 空状态不再只是普通文字块，而要具备风格化呈现

### 3.3 系统菜单页

系统菜单页要更像“配置控制台”，而不是简单表单页：

- 规则卡片像模块化控制单元
- source 卡片要突出当前启用状态和采集状态
- `手动执行采集` 需要成为一个明显的主操作区域
- 用户信息页要更像身份面板，而不是普通描述列表

### 3.4 legacy 页面

legacy 页面包括：

- `/history`
- `/reports/:date`
- `/control`

这些页面不重构结构，但要轻量接入新的视觉体系：

- 背景颜色与主站统一
- 按钮和卡片样式与统一站点不割裂
- 字体、边框和色彩保持同一套 token 体系

## 4. 主题机制与前端实现方案

### 4.1 主题机制

主题切换采用前端轻量方案：

- 在根节点上挂主题标识，例如 `data-theme="dark"` 或 `data-theme="light"`
- 深色和浅色分别拥有完整 CSS 变量集合
- 默认主题优先使用 `dark`
- 用户主动切换后，把选择结果写入浏览器本地存储
- 页面初始化时尽早读取并应用主题，减少闪烁

### 4.2 前端实现边界

本次实现以现有 SSR 架构为基础：

- `renderAppLayout.ts`
  - 增加主题切换 UI 和更强的壳层结构语义
- `site.css`
  - 重建为双主题 token + 赛博实验视觉系统
- `site.js`
  - 增加主题读取、切换、持久化和启动时应用逻辑
- `renderContentPages.ts`
  - 只做必要的语义结构增强，服务于样式表达
- `renderSystemPages.ts`
  - 只做必要的结构增强，支撑控制台视觉

本次改造允许适度增加类名、包裹层和语义结构，但不应改变现有路由、接口和交互契约。

### 4.3 持久化策略

主题偏好通过 `localStorage` 保存，理由如下：

- 当前产品是本地单用户站点
- 本地记忆即可满足需求
- 不需要为此改数据库或用户表
- 与现有 SSR + 少量浏览器增强模式最匹配

## 5. 组件级视觉策略

### 5.1 导航区

导航区要承担“控制台模块面板”的角色：

- 分组标题更像系统层级标记
- 当前激活菜单有明显高亮，而不是简单底色变化
- 底部主题切换器像一个控制模块，而不是普通 checkbox

### 5.2 卡片系统

卡片要区分为两类视觉语义：

- `内容卡片`
  - 更强调阅读和情报浏览
- `系统卡片`
  - 更强调控制、状态和配置

二者都要来自同一套视觉 token，但不能完全长得一样。

### 5.3 操作控件

按钮和交互控件要统一：

- 主操作按钮有更强发光或边框强调
- 收藏 / 点赞 / 点踩 这种次级操作要有清晰状态反馈
- 评分输入框、下拉框、文本域要统一成控制台风格

### 5.4 背景与氛围层

背景效果以 CSS 为主，避免复杂依赖：

- 渐变层
- 网格感或微纹理
- 轻量辉光
- 面板透明叠层

所有背景效果都必须服务于内容可读性，不能压过主体文字。

## 6. 测试与验收边界

### 6.1 必做验证

本阶段至少验证下面这些点：

- 统一站点页面能正常渲染
- 深色和浅色主题切换可用
- 刷新后主题偏好仍然保留
- 收藏、点赞、点踩、评分等现有内容交互不回归
- 系统菜单中的保存规则、切换 source、手动采集不回归
- legacy 页面仍可访问，且视觉与统一站点不明显割裂

### 6.2 验收范围

验收页面至少覆盖：

- `/`
- `/articles`
- `/ai`
- `/settings/view-rules`
- `/settings/sources`
- `/settings/profile`
- `/history`
- `/reports/:date`
- `/control`

### 6.3 非目标

本阶段不以“完整 UI 重构”作为验收标准，而以这些结果作为成功标准：

- 站点整体气质明显提升
- 主题切换稳定可用
- 统一站点和 legacy 页面视觉关系更协调
- 业务交互不回归
