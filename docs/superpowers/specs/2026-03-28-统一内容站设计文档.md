# HotNow 统一内容站设计文档

## 1. 目标与范围

### 1.1 当前基线

当前仓库已经具备一条稳定的日报工具链路：

`定时 / 手动触发 -> 拉取 RSS -> 抓取原文 -> 规则聚类 -> 生成 JSON/HTML 报告 -> 网页查看 -> SMTP 发邮件`

这条链路已经能稳定服务“单机日报生成”场景，但还不是一个正式的内容站点。现阶段要做的不是推翻现有实现，而是在保留 RSS、报告和邮件这些基础能力的前提下，把项目升级成一个统一登录、统一导航、统一内容池的正式网页项目。

### 1.2 新目标

HotNow 下一阶段的产品目标是：

`构建一个单用户、统一站点、以 RSS 聚合为底座的 AI/科技内容浏览与管理应用`

其中：

- RSS 抓取是内容入口
- HTML 报告和邮件发送是系统输出能力
- 网站本身是主产品形态
- 内容浏览、交互操作和系统管理都在同一个站点里完成

### 1.3 本阶段已确认决策

- 站点形态为单用户统一站点，不拆前台和后台项目
- 登录后可直接看到内容菜单和系统菜单
- 数据存储采用 `SQLite`
- `热点资讯`、`热门文章`、`最新 AI 消息` 来自同一个内容池的三种视图
- 三种视图按内容标签和规则驱动，不按来源硬编码划分
- 保留并继续支持现有报告生成、历史查看和邮件发送能力
- RSS source 层需要从单一 `juya` 扩展为可注册适配器的结构
- 第一阶段先支持：
  - `juya`
  - `openai`
  - `google_ai`
  - `techcrunch_ai`

### 1.4 本阶段明确不做

- 多用户体系
- 评论、关注、私信、分享等社交能力
- 推荐算法和个性化内容编排
- 复杂角色权限模型
- 可视化规则 DSL 编辑器
- 搜索引擎级全文检索
- 独立 App / 小程序
- 云原生多实例部署

## 2. 产品定位与信息架构

### 2.1 产品定位

HotNow 不是纯粹的新闻阅读器，也不是传统后台管理系统。它的核心定位是：

- 用统一内容池承接来自 RSS 的科技与 AI 内容
- 通过规则和标签把内容组织成不同浏览视图
- 允许单用户对内容做收藏、反馈和评分
- 允许单用户直接管理采集策略、筛选策略和系统配置

### 2.2 一级菜单结构

统一站点中的一级菜单分成两组，但处于同一套导航结构中：

#### 内容菜单

- `热点资讯`
- `热门文章`
- `最新 AI 消息`

#### 系统菜单

- `筛选策略`
- `数据迭代收集`
- `当前登录用户`

### 2.3 菜单语义

#### 热点资讯

偏时效性和热度。优先展示热度高、更新快、适合作为情报扫描入口的内容。

#### 热门文章

偏阅读价值和完整度。优先展示正文质量更完整、值得进一步点开阅读和收藏的文章内容。

#### 最新 AI 消息

偏 AI 相关性和新鲜度。优先展示 AI 标签强、发布时间新的内容。

#### 筛选策略

负责配置三类视图的筛选阈值、排序权重和标签策略，但第一阶段不提供复杂表达式编辑器。

#### 数据迭代收集

负责管理已注册数据源、当前启用源、抓取频率、最近执行结果和手动采集。

#### 当前登录用户

负责展示单个管理员账号的基础信息、登录状态和后续可扩展的账号设置入口。

## 3. 统一内容池与视图规则

### 3.1 统一内容池

所有 RSS 内容先统一落到一个标准内容对象 `content_item`。任何 source adapter 输出的结果，都必须先映射成统一字段，然后才能进入后续规则链路、内容列表和报告系统。

统一内容池至少承载以下信息：

- 标题
- 摘要
- 正文摘录
- 原始链接
- 来源名称
- 来源类型
- 发布时间
- 内容类型
- AI 相关性
- 热度分数
- 标签集合
- 抓取状态
- 创建时间和更新时间

### 3.2 三种视图的实现方式

三种内容菜单不维护三套独立数据，而是对同一个内容池应用不同规则：

- `热点资讯`
  - 偏高 `hot_score`
  - 偏强时效
  - 偏资讯型内容

- `热门文章`
  - 偏正文完整度
  - 偏收藏和评分表现
  - 偏深度阅读型内容

- `最新 AI 消息`
  - 偏高 `ai_relevance`
  - 偏最新发布时间
  - 偏 AI 标签命中内容

### 3.3 规则驱动原则

第一阶段采用“字段 + 固定规则函数”的方式，而不是 DSL：

- 内容入库时补齐基础标签和数值字段
- 菜单页使用固定规则函数进行筛选和排序
- 系统配置页只修改规则参数，不直接修改代码表达式

这种方式的优点是：

- 比每个菜单手写一套散乱 SQL 更稳定
- 比 DSL 更轻，适合第一阶段
- 后续扩展新菜单、新标签和新源时更容易维护

## 4. 页面结构与交互

### 4.1 页面骨架

整个站点采用一个统一应用壳层：

- 左侧或顶部导航区域
- 页面标题和说明区域
- 主内容工作区
- 当前登录用户入口

第一阶段不把它做成双系统，而是做成一个统一工作台。

### 4.2 内容菜单页面结构

三个内容菜单页面沿用同一套结构，以降低实现复杂度并保持一致体验：

#### 筛选栏

- 时间范围
- 来源过滤
- 标签过滤
- 排序方式
- 收藏状态过滤

#### 内容列表

每条内容卡片展示：

- 标题
- 摘要
- 来源
- 发布时间
- 标签
- 热度/相关度摘要
- 收藏按钮
- 点赞按钮
- 点踩按钮
- 评分入口

#### 详情区

点击内容后展示：

- 更完整的摘要或正文摘录
- 原始链接
- 来源信息
- 采集状态
- 评分结果摘要

第一阶段可以采用“列表 + 侧边详情”或“列表 + 详情页”的轻量模式，不要求复杂阅读器形态。

### 4.3 系统菜单页面结构

#### 筛选策略页

展示每个内容菜单对应的规则参数：

- 热度阈值
- AI 相关性阈值
- 收藏和评分权重
- 允许展示的标签
- 排序优先级

#### 数据迭代收集页

展示 source adapter 注册情况与运行状态：

- 当前启用 source
- 已知 source 列表
- 各 source 的 RSS 地址
- 最近抓取时间
- 最近抓取状态
- 手动执行采集按钮

#### 当前登录用户页

展示：

- 用户名
- 登录状态
- 基础资料
- 账号设置入口

第一阶段不扩展为多用户管理列表。

### 4.4 视觉方向

网站视觉不应继续保持“报告查看器”的简单形态，而应更接近“科技内容工作台”：

- 内容区有明显的信息层级
- 内容卡片突出标题、摘要和来源
- 管理页面与内容页面共享同一套视觉语言
- 避免做成纯表格后台，也避免做成过于花哨的资讯流

## 5. 收藏、反馈与评分系统

### 5.1 基础交互

每条内容都支持以下交互：

- 收藏
- 点赞
- 点踩

其中点赞和点踩互斥，同一用户同一时间只能保留一种反馈状态。

### 5.2 评分系统

评分系统需要具备真正可落地的结构，但第一阶段不把维度写死在 UI 中。设计要求如下：

- 评分能力必须存在
- 评分维度可以配置
- 页面读取评分维度配置后动态渲染评分表单
- 默认维度可以由系统初始化生成，但后续必须可替换

### 5.3 评分接口边界

评分模块至少需要这几类能力：

- 读取当前生效的评分维度定义
- 提交某条内容的评分结果
- 读取某条内容的评分汇总
- 为后续替换维度保留兼容边界

第一阶段不做复杂评分历史分析，但不能把结构设计成只支持单一总分值。

## 6. 数据模型设计

### 6.1 存储选型

第一阶段采用 `SQLite`，原因如下：

- 单用户场景下实现和维护成本最低
- 足以支撑内容、反馈、评分和配置类数据
- 更贴合当前单体 Fastify 项目
- 不需要为了实时协作能力提前引入额外复杂度

### 6.2 核心实体

第一阶段至少需要以下实体或表：

#### content_sources

记录已注册的 RSS source：

- `id`
- `kind`
- `name`
- `rss_url`
- `enabled`
- `is_active`
- `fetch_interval_minutes`
- `last_fetched_at`
- `last_fetch_status`

说明：

- 第一阶段允许预置多个 source
- 运行时仅允许一个 `active` source 参与正式采集
- 这个约束与当前“先保证单源稳定、后续再扩并发多源”保持一致

#### content_items

记录统一内容池中的内容项：

- `id`
- `source_id`
- `external_id`
- `title`
- `summary`
- `excerpt`
- `url`
- `source_name`
- `published_at`
- `content_type`
- `ai_relevance`
- `hot_score`
- `tags_json`
- `fetch_status`
- `created_at`
- `updated_at`

#### content_feedback

记录收藏与正负反馈：

- `id`
- `content_item_id`
- `is_favorited`
- `reaction`
- `updated_at`

其中 `reaction` 取值为：

- `like`
- `dislike`
- `none`

#### rating_dimensions

记录评分维度定义：

- `id`
- `key`
- `label`
- `description`
- `weight`
- `enabled`
- `sort_order`

#### content_ratings

记录内容评分结果：

- `id`
- `content_item_id`
- `dimension_key`
- `score`
- `updated_at`

#### view_rule_configs

记录三类内容菜单的规则参数：

- `id`
- `view_key`
- `config_json`
- `updated_at`

#### user_profile

记录单用户管理员信息：

- `id`
- `username`
- `display_name`
- `password_hash`
- `last_login_at`
- `updated_at`

#### collection_runs

记录采集和导入执行结果：

- `id`
- `source_id`
- `trigger`
- `started_at`
- `finished_at`
- `status`
- `summary_json`

#### digest_reports

记录日报产物索引：

- `id`
- `report_date`
- `report_json_path`
- `report_html_path`
- `mail_status`
- `generated_at`

### 6.3 兼容原则

现有基于文件的报告产物目录结构应继续保留：

- `data/reports/<YYYY-MM-DD>/report.json`
- `data/reports/<YYYY-MM-DD>/report.html`
- `data/reports/<YYYY-MM-DD>/run-meta.json`

SQLite 在新架构中的职责是为站点能力提供状态和查询能力，而不是立即替代现有报告文件产物。

## 7. RSS 适配器与采集策略

### 7.1 适配器模式

source 层从“单一解析函数”升级为“适配器注册表”：

- 每个 source 有明确 `kind`
- `loadLatestIssue` 不再直接绑定 `parseJuyaIssue`
- 先拉取 XML，再根据 `source.kind` 分发到对应适配器
- 适配器输出统一内容输入结构

### 7.2 第一阶段支持的 source

第一阶段需支持：

- `juya`
- `openai`
- `google_ai`
- `techcrunch_ai`

设计要求：

- `juya` 解析器保持当前稳定行为，不允许回归
- `openai`、`google_ai`、`techcrunch_ai` 可复用一个标准 article RSS 解析基座
- 每个 source 仍通过显式注册接入，避免隐式按域名猜测

### 7.3 采集策略

第一阶段采用“预置多源，但正式采集只激活一个源”的约束：

- 数据迭代收集页展示多个已注册 source
- 管理员可以切换当前 active source
- 定时采集和手动采集都只对 active source 生效
- 后续如果要升级成多源并发采集，可在此基础上扩展

这个约束可以同时满足：

- 当前解析器稳定性要求
- 后续多源扩展的结构准备
- 统一内容池的长期演进方向

## 8. 报告与邮件的保留策略

### 8.1 报告角色重定义

在新站点中，日报报告不再是唯一主入口，但仍然保留为重要系统输出：

- 对外是内容摘要产物
- 对内是站点运行结果的日级快照
- 对邮件发送是复用模板来源

### 8.2 保留能力

以下现有能力必须保留：

- 每日定时生成报告
- 手动生成报告
- 历史报告查看
- HTML 报告落盘
- JSON 报告落盘
- SMTP 邮件发送

### 8.3 新站点中的位置

报告系统应被视为站点中的一个系统能力，而不是整个产品唯一形态。站点应允许用户既能浏览统一内容池，也能查看每日聚合后的报告结果。

## 9. 接口边界与模块职责

### 9.1 后端模块边界

第一阶段建议按下面的边界组织模块：

- `auth`
  - 单用户登录和会话管理
- `sources`
  - source 注册、适配器分发、采集执行
- `content`
  - 内容入库、标签、查询和详情
- `feedback`
  - 收藏、点赞、点踩
- `ratings`
  - 评分维度与评分提交
- `view-rules`
  - 三类内容菜单的规则配置
- `reports`
  - 报告生成、报告索引、历史报告
- `mail`
  - 邮件发送

### 9.2 接口分类

虽然第一阶段可以保持单体应用，但逻辑边界应按接口思路设计：

- 登录与用户信息接口
- 内容列表与详情接口
- 收藏与反馈接口
- 评分维度与评分提交接口
- 视图规则读取与保存接口
- source 配置与采集触发接口
- 报告查看与历史接口

这样可以避免页面层和存储层直接耦合，也为后续前端增强留出空间。

## 10. 第一阶段落地边界

### 10.1 第一阶段必须交付

- 单用户登录后的统一站点壳层
- 三个内容菜单：
  - 热点资讯
  - 热门文章
  - 最新 AI 消息
- 三类基础交互：
  - 收藏
  - 点赞 / 点踩
  - 评分
- 三个系统菜单：
  - 筛选策略
  - 数据迭代收集
  - 当前登录用户
- SQLite 持久化
- source adapter 注册表
- `juya / openai / google_ai / techcrunch_ai` 解析支持
- 保留定时采集、手动采集、报告生成和邮件发送

### 10.2 第一阶段不交付

- 多用户
- 评论和社交
- 推荐系统
- 复杂权限
- 高级可视化规则编辑器
- 多源并发正式采集
- 云端部署体系

## 11. 风险与设计约束

### 11.1 风险

- 现有代码以日报工具为中心，升级为统一站点时需要控制改动边界，避免一次性重写
- 不同 RSS feed 结构差异较大，source adapter 测试必须前置
- 单用户站点虽然简单，但登录、配置保存和交互状态一旦写死，后续扩展会很难
- 评分维度如果一开始偷懒做成单分值，后面会直接卡住演进

### 11.2 设计约束

- 不得破坏现有 `juya` 报告链路
- 不得移除当前报告文件产物
- 不得把敏感信息写入仓库
- 不得让 UI 结构先行把后端模型绑死
- 新站点必须建立在“统一内容池 + 可扩展 source adapter + 可配置规则”三根主轴上

## 12. 最终结论

HotNow 下一阶段应按以下原则实施：

- 保留现有日报能力作为基础设施
- 把项目升级为单用户统一站点
- 用 `SQLite` 承接内容、交互、评分和配置
- 用统一内容池支撑三种内容视图
- 用 source adapter 注册表支撑 RSS 扩展
- 用系统菜单承接规则配置、数据采集和用户信息

这使得项目从“可跑的日报工具”升级成“可持续演进的正式内容产品”，同时不牺牲当前已跑通的稳定链路。
