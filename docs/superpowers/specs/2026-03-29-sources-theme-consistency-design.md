# 数据迭代收集卡片主题一致性 Design

## 背景

`/settings/sources` 页面已经完成了“即时操作”和“数据源状态”的结构分区，用户能明显看出：

- `手动执行采集`
- `手动发送最新报告`
- `source` 状态卡

属于两类不同对象。

但当前卡片样式还有一个明显问题：

- 深色主题下，操作卡和 source 卡的视觉方向基本成立
- 浅色主题下，这几类卡片仍保留了偏深色的背景叠层和过暗的半透明色块

结果就是：

- 页面整体是浅色
- 卡片却像深色控制台浮层
- 部分标题、说明和元信息对比度不足
- 用户会感觉主题混乱，不像一个真正完成的浅色模式

当前问题不是“浅色主题 token 没切换”，而是 `/settings/sources` 里若干卡片样式写死了偏深色的背景值，导致它们没有真正跟随主题语义。

## 目标

### 核心目标

让 `/settings/sources` 页满足下面的直觉规则：

- 浅色模式就是完整的浅色模式
- 深色模式就是完整的深色模式
- 两种模式下都保留“操作卡”和“数据源卡”的层级差异，但不再出现主题串色

### 用户感知目标

用户在切换主题时，不应该感觉是：

- 页面换成浅色了，但卡片还是深色
- 或者背景变亮了，但正文和说明文字仍然发灰发糊

而应该感觉是：

- 整个页面和卡片一起进入新的主题体系
- 仍然能区分“这是操作卡”“这是数据源卡”
- 但不会因为跨主题串色导致视觉混乱

## 设计原则

### 1. 卡片语义跟随主题，不写死深色外观

`/settings/sources` 用到的三类卡片：

- `操作主卡`
- `操作次卡`
- `数据源卡`

都应该是“语义卡片”，而不是“写死某种深色背景的卡片”。

换句话说：

- 先定义卡片的角色
- 再让深色主题和浅色主题分别给这个角色一套正确的外观

### 2. 浅色主题要像真正的浅色控制台

浅色模式下：

- `操作主卡`
  - 仍然最醒目
  - 但底色应是浅的、干净的、有冷蓝高光的亮面卡
- `操作次卡`
  - 比主卡更克制，但仍属于动作区
- `数据源卡`
  - 应该是清晰的浅底信息卡
  - 标题、元信息、链接和状态要有足够对比

不能再出现大面积深蓝/深黑底色漂浮在浅色页面中的效果。

### 3. 深色主题保留控制台层级，但提升可读性

深色模式下不需要推翻当前方向，只做收敛：

- 保留操作卡的发光感和优先级
- source 卡维持克制的信息卡语义
- 标题、说明、元信息的对比度要够
- 避免过重的暗色叠层把文字压暗

## 样式边界

### 会调整的对象

主要调整下面三类卡：

- `.system-card--operation.system-card--operation-primary`
- `.system-card--operation.system-card--operation-secondary`
- `.system-card--inventory`

以及它们内部相关文字层：

- `.system-card-title`
- `.system-card-meta`
- `.system-card-kicker`
- `code`
- 链接文本
- 状态文本

### 不改的部分

本轮不做下面这些事：

- 不改 `/settings/sources` 的 section 结构
- 不改按钮行为
- 不改 `data-system-action`
- 不改采集或发信逻辑
- 不改其他菜单页的卡片系统
- 不改全站主题切换机制

## 实现约束

### 优先方案

优先把当前写死的深色背景值替换成“跟随主题的语义样式”，而不是只对浅色模式打补丁。

更具体地说：

- 不再直接依赖固定深色 `rgba(...)` 作为主背景
- 改为使用当前主题 token 组合出不同角色卡片
- 只在必要时为 light/dark 各自写角色差异化覆盖

### 允许的技术做法

- 在 `site.css` 新增少量针对 light/dark 的卡片语义覆盖
- 如果现有 token 不够表达，可以补少量新 token
- 如必须增强结构语义，可少量补 class，但不要改页面行为

## 验收标准

### 浅色模式

- 操作卡和 source 卡都属于浅色体系
- 标题、说明、元信息清晰可读
- 不再出现“浅色页面 + 深色大卡片”的混搭感
- 仍能看出操作卡高于 source 卡

### 深色模式

- 操作卡仍然更强
- source 卡仍然更克制
- 文案对比度充足，不发糊
- 整体保持控制台感，但不脏不闷

### 回归要求

- `/settings/sources` 的结构分区保持不变
- 现有按钮和状态文案行为不变
- 最相关测试和构建仍需通过
