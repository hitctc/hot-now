# HotNow 内容优先级与壳层收敛设计文档

## 1. 目标与范围

### 1.1 当前基线

当前 HotNow 已经具备：

- 单用户登录的统一站点壳层
- `热点资讯 / 热门文章 / 最新 AI 消息` 三个内容菜单
- `筛选策略 / 数据迭代收集 / 当前登录用户` 三个系统菜单
- RSS 抓取、原文提取、统一内容池、HTML 报告和邮件发送
- 当前 source adapter：
  - `juya`
  - `openai`
  - `google_ai`
  - `techcrunch_ai`

但当前实现还有三个明显问题：

- 顶部 header 占用了过多可视空间，和左侧导航职责重叠
- 评分仍然依赖人工 1-5 分录入，不符合“快速判断值不值得看”的目标
- source 仍然以“单一当前启用 source”为中心，难以稳定拿到“最新、一手、高质量”的科技资讯

### 1.2 本阶段新目标

本阶段要把 HotNow 从“可用的内容站”推进成“面向单用户的科技资讯控制台”，目标是：

`稳定聚合多个可信 RSS 源中的最新科技 / AI 内容，并用系统自动评分帮助用户优先看到最新、一手、高质量的资讯与文章。`

### 1.3 本阶段已确认决策

- 页面壳层去掉顶部 header，把当前页面信息收回到左侧侧边栏
- 当前登录用户信息和退出操作移到左侧侧边栏底部
- 页面评分从“用户手工评分”改为“系统自动百分制评分”
- 用户不再手工给文章打分；页面主要展示系统生成的 `0-100` 分值
- 内容采集从“单一当前启用 source”改成“多源同时采集并汇总”
- source 组合采用：
  - 官方源
  - 快速科技媒体
  - AI 垂直媒体
- 新鲜度窗口以最近 `7` 天为主，超过 `7` 天内容明显降权
- 三个内容菜单语义明确为：
  - `热点资讯`：热度和时效并重
  - `热门文章`：深度和质量优先
  - `最新 AI 消息`：AI 相关性和发布时间优先

### 1.4 本阶段明确不做

- 多用户体系
- 大模型打分或大模型排序
- 自动生成深度摘要或评论型总结
- 复杂事件聚类或跨天趋势分析
- 在线新增任意自定义 source adapter 的后台能力
- 推荐算法和个性化推荐系统

## 2. 壳层与页面结构调整

### 2.1 总体方向

统一站点应从“左侧导航 + 顶部 header + 主内容区”收敛为“单一侧边控制台 + 纯内容工作区”的结构。

目标不是减少信息，而是把壳层信息放到更合理的位置，让右侧主区域尽量给内容列表和系统卡片留空间。

### 2.2 左侧侧边栏职责

左侧侧边栏将承担以下职责：

- 品牌与站点定位
- 当前页面标题与简短说明
- 内容菜单导航
- 系统菜单导航
- 主题切换
- 当前登录用户信息
- 退出登录按钮

### 2.3 顶部区域调整

当前顶部 `shell-header` 将被移除，不再保留独立的一整行页面标题区。

替代方案：

- 当前页面标题与说明展示在侧边栏上部，与品牌区和导航相邻
- 右侧主区域从页面顶部直接开始渲染内容，不再被 header 占据高度

### 2.4 用户区位置

当前登录用户区从顶部右侧移到底部侧边栏，与主题切换同属“个人与界面控制区域”。

底部区域的顺序建议为：

1. 主题切换
2. 当前登录用户摘要
3. 退出登录按钮

### 2.5 内容页与系统页的空间目标

完成壳层收敛后：

- 内容页能展示更多首屏卡片
- 系统页中 JSON 策略、source 状态和操作按钮有更大可用空间
- 页面整体更像“情报控制台”，而不是“有后台 header 的普通站点”

## 3. 自动评分系统

### 3.1 评分目标

新的分数不是“用户主观喜好分”，而是系统对内容价值的快速判断：

`这条内容现在是否值得优先查看，以及值得投入多少注意力。`

第一阶段统一使用 `0-100` 的百分制分数，命名为：

- `contentScore`

### 3.2 评分信号

`contentScore` 由以下几类规则信号组成：

#### 新鲜度

- 最近 `24` 小时内容获得最高新鲜度加成
- 最近 `7` 天内容按时间递减
- 超过 `7` 天内容明显降权

#### 来源质量

- 官方源具有较高可信度基础分
- 权威科技媒体和 AI 垂直媒体获得中高基础分
- 聚合型或二手整理型来源只作为补充信号，不主导总分

#### 内容完整度

- 有明确发布时间
- 有摘要
- 有可抓取正文
- 正文长度合理、不是残缺页或极短页

#### AI 相关性

- 标题、摘要、正文命中 AI / LLM / GPT / agent / model / 大模型 / 智能体等关键词时获得相关分
- 在 `最新 AI 消息` 视图中，该部分权重更高

#### 热点信号

- 多个 source 提到相近主题时加分
- 命中高价值事件词时加分，例如：
  - 发布
  - 开源
  - 融资
  - 上线
  - 更新
  - 收购

### 3.3 页面呈现

页面里不再展示“评分表单”或用户打分下拉框。

每条内容卡片改为展示：

- 系统分数，例如 `86 / 100`
- 少量可解释标签，例如：
  - `24h 内`
  - `官方源`
  - `正文完整`
  - `多源共振`

### 3.4 用户交互保留范围

本阶段保留：

- 收藏
- 点赞
- 点踩

本阶段移除：

- 手工评分录入表单
- 评分维度选择器

点赞 / 点踩继续保留，但语义收窄为“用户反馈信号”，用于后续排序迭代，不在本阶段直接替代系统评分。

## 4. 三类内容菜单的排序语义

### 4.1 统一内容池原则

三个内容菜单仍然来自同一个统一内容池，不拆成三套独立采集和存储系统。

差异只体现在排序和权重上。

### 4.2 热点资讯

`热点资讯` 的排序应优先考虑：

- 新鲜度
- 热点信号
- 多源共振
- 来源质量

它的目标是快速回答：

`今天科技圈和 AI 圈最值得先扫一眼的内容是什么。`

### 4.3 热门文章

`热门文章` 的排序应优先考虑：

- 内容完整度
- 来源质量
- 正文长度和深读价值
- 新鲜度

它的目标是回答：

`哪些内容值得花更完整的时间阅读。`

### 4.4 最新 AI 消息

`最新 AI 消息` 的排序应优先考虑：

- AI 相关性
- 发布时间
- 来源质量
- 热点信号

它的目标是回答：

`最近真正新的 AI 动向、产品、模型和行业更新是什么。`

## 5. 多源采集策略

### 5.1 采集模式调整

当前“单一当前启用 source”的模式将升级为“多源同时采集”。

新模式下：

- 系统维护一组 source 启用状态
- 每次采集任务会遍历所有启用 source
- 各 source 内容统一入同一个内容池
- 页面查询与报告生成都基于汇总后的统一内容池工作

### 5.2 source 组合原则

第一阶段 source 组合采用混合策略：

- 官方源
- 快速科技媒体
- AI 垂直媒体

内置 source 至少包括当前已有的：

- `OpenAI`
- `Google AI`
- `TechCrunch AI`
- `Juya AI Daily`

同时应预留后续继续补充：

- `VentureBeat AI`
- 科技社区型热榜信号源

### 5.3 source 权重

不同 source 不应被视为完全等价。

source 应具备至少以下元信息：

- `sourceType`
- `sourcePriority`
- `freshnessExpectation`
- `qualityBaseline`

其中：

- 官方源更偏一手可信度
- 媒体源更偏时效与热度覆盖
- 聚合源更偏热点线索补充

### 5.4 菜单与 source 的关系

三个内容菜单不再绑定某个特定 source。

例如：

- `最新 AI 消息` 不再等于 “OpenAI 源内容”
- `热门文章` 不再等于 “媒体长文源”
- `热点资讯` 不再等于 “聚合日报”

它们都从同一个多源内容池中筛选，只是排序偏好不同。

## 6. 系统菜单调整

### 6.1 筛选策略

`筛选策略` 页面从“编辑简单 JSON 配置”升级为“管理内容排序权重的控制面板”。

第一阶段至少允许配置：

- 新鲜度权重
- 来源质量权重
- 内容完整度权重
- AI 相关性权重
- 热点信号权重
- 每个视图的展示数量上限

### 6.2 数据迭代收集

`数据迭代收集` 页面从“切换当前启用 source”升级为“管理多源采集状态”。

每个 source 至少展示：

- 名称
- RSS 地址
- 是否启用
- 最近抓取时间
- 最近抓取状态

页面顶部保留：

- 手动执行一次全量采集

### 6.3 当前登录用户

`当前登录用户` 页保持轻量，不扩成复杂账户中心。

由于当前用户信息会移入侧边栏底部，该页面后续可以变成更轻的账号详情说明页，而不再承担主要身份展示职责。

## 7. 数据与实现边界

### 7.1 数据模型调整方向

本阶段至少需要引入或补齐以下持久化语义：

- 每条内容的系统分数字段或可稳定重建的评分输入字段
- 每个 source 的启用状态
- 每个 source 最近一次采集状态
- 视图级排序权重配置

### 7.2 兼容原则

本阶段不应打断已有能力：

- HTML 报告生成
- 历史报告查看
- SMTP 邮件发送
- 现有 unified shell 登录能力

### 7.3 迁移原则

本阶段属于“统一内容站的第二轮优化”，不是重写项目。

因此应优先：

- 复用现有 SQLite
- 复用现有 source adapter 注册方式
- 复用现有 unified shell 路由结构
- 复用现有采集任务主链路

## 8. 验收标准

本阶段完成后，至少满足以下结果：

- 顶部 header 被移除，页面标题和用户信息都收进左侧侧边栏
- 页面首屏可用空间明显增加
- 内容卡片不再出现人工评分表单
- 每条内容展示系统自动生成的百分制分数
- 采集任务支持对多个启用 source 同时执行
- `最新 AI 消息` 不再只受单一 `OpenAI` source 控制
- 最近 `7` 天内容在三个视图中都显著优先于旧内容
- 系统菜单中的 `数据迭代收集` 能展示多 source 启用与抓取状态
- 现有报告、历史页、登录与邮件能力不回归
