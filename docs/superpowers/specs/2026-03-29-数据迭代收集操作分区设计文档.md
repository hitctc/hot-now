# 数据迭代收集操作分区 Design

## 背景

当前统一站点的 `/settings/sources` 页面已经包含三类能力：

- `手动执行采集`
- `手动发送最新报告`
- 多个 `source` 的启用/停用与状态查看

虽然“操作卡片”和“数据源卡片”在语义上已经分离，但当前页面把它们连续堆在同一组 `system-card` 中，视觉权重和结构节奏接近，用户容易把“执行动作”和“查看/管理数据源状态”读成同一类对象。

这会带来两个直接问题：

- 用户进入页面后，难以快速区分“现在可以执行什么动作”和“系统当前有哪些数据源”。
- `手动执行采集`、`手动发送最新报告` 的优先级不够高，和单个 source 的启停按钮抢注意力。

本轮目标不是新增功能，而是把 `/settings/sources` 改造成一个更清楚的控制台页面，让“操作面板”和“数据源列表”在结构和视觉上都明显分层。

## 目标

### 页面结构

`/settings/sources` 调整为两个明确的 section：

1. `即时操作`
   - 放在页面上方
   - 只包含两张操作卡：
     - `手动执行采集`
     - `手动发送最新报告`
2. `数据源状态`
   - 放在操作面板下方
   - 展示所有 source 卡片及其启用状态、最近抓取时间、最近抓取状态、启停按钮

### 用户认知目标

用户进入页面后，应当在几秒内形成下面的阅读顺序：

1. 先看到“我现在可以做的动作”
2. 再看到“系统当前接入和启用了哪些数据源”

即使不仔细看文案，也应该能从布局和视觉上分辨出这两类卡片不是同一层对象。

## 视觉语义

### 即时操作区

即时操作区定位为高优先级控制面板：

- 有独立 section 标题与说明文字
- 两张操作卡使用更强的视觉强调
- 卡片体积、留白和按钮权重高于 source 卡片
- 操作卡需要额外展示“影响范围”或“行为说明”

两张操作卡内部仍然保持区分：

- `手动执行采集`
  - 作为更强的主操作
  - 更强调“抓取、聚类、更新报告”
- `手动发送最新报告`
  - 作为次级操作
  - 更强调“重发最新产物，不重新抓取”

### 数据源状态区

数据源区定位为状态清单：

- 有独立 section 标题与说明文字
- 整体风格更克制，强调可扫描性
- source 卡片保留统一列表秩序，不与主操作抢视觉中心
- 启用/停用按钮仍可见，但权重低于操作卡主按钮

## 非目标

本轮不做下面这些事：

- 不改 `POST /actions/run`
- 不改 `manual-send-latest-email` 的接口行为
- 不改 source 启用/停用逻辑
- 不改数据库结构
- 不改页面路由
- 不重做 `/settings/view-rules` 或 `/settings/profile`

## 实现边界

### 会修改的文件

- `src/server/renderSystemPages.ts`
  - 把 `/settings/sources` 拆成两个分区容器
  - 给操作卡和 source 卡分别提供更明确的 section 结构
- `src/server/public/site.css`
  - 新增操作区和数据源区的布局与视觉规则
  - 强化操作卡，降低 source 卡视觉噪音
- `tests/server/systemRoutes.test.ts`
  - 断言页面中存在两个明确区域，而不是只有连续卡片

### 不修改的行为

- 所有表单 `data-system-action` 保持不变
- 前端 `site.js` 的动作分发不因为本轮结构调整而变更协议
- source 状态刷新和按钮回写逻辑保持现状

## 验收标准

### 结构

- `/settings/sources` 页面出现两个清晰 section：
  - `即时操作`
  - `数据源状态`
- 两张操作卡只出现在“即时操作”区
- source 卡片只出现在“数据源状态”区

### 视觉

- 操作卡视觉优先级明显高于 source 卡
- 不需要依赖阅读长文案，也能看出两类卡片不同
- 页面阅读顺序自然，从“动作”到“状态”

### 回归

- 手动采集按钮仍可触发原有动作
- 手动发送最新报告按钮仍可触发原有动作
- source 启用/停用按钮仍可用
- 现有页面测试和最相关的 `/settings/sources` 渲染测试继续通过
