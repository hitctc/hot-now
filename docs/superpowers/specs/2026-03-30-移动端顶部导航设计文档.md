# HotNow 移动端顶部导航 Design

## 背景

当前 unified shell 在桌面端使用左侧侧边栏承载 `内容菜单`、`系统菜单`、品牌块、当前页面摘要、主题切换和账号区。这套结构在桌面端成立，但在移动端直接压缩后会出现两个问题：

- 顶部区域被整块导航卡片占满，滚动首屏时内容入口不够直接。
- `内容菜单` 与 `系统菜单` 虽然都可见，但交互层级混在一起，用户为了切换高频内容页要先经过一段较高的卡片区。

前一轮尝试把两组菜单整体做成移动端置顶浮层，虽然功能可用，但顶部高度过高，本质上还是“把桌面侧边栏缩小后搬到顶部”，没有真正转成移动端导航模式。

## 目标

### 核心目标

- 在移动端把高频导航切换收口到一个更符合手持设备习惯的顶部导航模式。
- 让 `内容菜单` 保持一级导航的高频可达性。
- 让 `系统菜单` 保持可发现，但不再和内容导航抢首屏高度。
- 明显压缩移动端顶部导航占高，使正文更早进入视口。

### 用户感知目标

- 用户进入任意内容页时，能立刻在顶部看到当前可切换的内容入口。
- 用户需要系统功能时，能明确找到 `系统菜单` 按钮，而不是猜测它藏在哪。
- 用户在长页面滚动过程中，仍然能方便切换内容页或打开系统入口。

### 非目标

- 不改桌面端统一壳层布局。
- 不改现有路由、页面语义或 active 高亮逻辑。
- 不引入全屏抽屉、复杂动画或多级菜单结构。
- 不调整内容页和系统页自身的数据读取与渲染逻辑。

## 设计结论

### 主方向

移动端采用 `内容 Tabs + 系统抽屉入口` 的结构：

- 顶部只保留一条低高度 `sticky` 浮层导航。
- 浮层左侧为 `内容菜单` tabs：`热点资讯 / 热门文章 / 最新 AI 消息`。
- 浮层右侧为一个明确写着 `系统菜单` 的按钮。
- 点击 `系统菜单` 后，在导航条下方展开一个轻量抽屉，显示 `筛选策略 / 数据迭代收集 / 当前登录用户` 三个系统入口。
- 抽屉默认关闭，只在用户主动打开时占用额外高度。

### 为什么采用这个方向

- 它把高频的内容切换保留为始终可见的一级导航。
- 它把低频但仍需强调的系统功能保留为明确入口，而不是藏在图标或更多菜单里。
- 它显著降低顶部占高，避免当前“导航功能有了，但正文首屏被压缩”的问题。
- 它比双层导航更节省高度，比单行 tabs + 更多菜单更能保留 `系统菜单` 的存在感。

### 明确不采用的方向

- 不采用“顶部双层常驻导航”，因为总体高度仍偏大。
- 不采用“整块侧边栏变顶部卡片”的方案，因为那只是桌面布局的缩小版。
- 不采用纯图标系统按钮，因为可发现性不够，且不利于区分 `内容菜单` 与 `系统菜单`。

## 布局规范

### 断点

- `max-width: 900px`：启用移动端顶部导航模式。
- `max-width: 640px`：继续压缩字号、padding 与 tabs 宽度。

### 顶部导航条

- 导航条整体使用 `sticky`，固定在页面顶部。
- 视觉形式为低高度、浅纸感、半透明浮层，而不是厚重卡片。
- 目标整体高度控制在 `52px` 到 `60px`。
- 导航条与正文之间保留轻微阴影和留白，确保悬浮关系清楚。

### 内容菜单 Tabs

- 3 个内容入口横向排列。
- tabs 保持单行，不换行。
- tabs 区域允许横向滚动，优先保证点击区域和文案可读性，而不是强行压缩。
- 当前内容页使用蓝色实心高亮，非当前项使用轻底色 pill。

### 系统菜单按钮

- 位于导航条右侧，与内容 tabs 同高。
- 按钮文案固定为 `系统菜单`，不用纯图标代替。
- 默认使用较轻的橙色辅助语义，打开时进入明显高亮态。

### 系统抽屉

- 抽屉展开位置在导航条正下方。
- 展开后仍然依附于顶部导航，而不是占据整屏。
- 只承载 3 个入口：
  - `筛选策略`
  - `数据迭代收集`
  - `当前登录用户`
- 抽屉默认单列排列，按钮面积优先保障可点按。
- 首屏占高控制在 `120px` 到 `168px`，不额外堆放描述文案。

## 交互规范

### 内容 Tabs

- 点击后直接跳转对应内容页。
- 如果系统抽屉处于打开状态，点击内容 tab 时先关闭抽屉，再执行跳转。
- 当前 tab 的 active 态由现有路由元数据继续驱动，不新增状态源。

### 系统菜单按钮

- 默认关闭。
- 点击一次打开抽屉，再点一次关闭。
- 抽屉打开时按钮进入橙色高亮态。
- 点击抽屉外部区域时关闭抽屉。

### 系统抽屉入口

- 点击后直接跳转目标系统页。
- 跳转前不需要额外确认。
- 跳转后抽屉不保留展开态。

### 公开访问模式

- 如果当前页面本来就隐藏系统菜单，则移动端顶部也不渲染 `系统菜单` 按钮与抽屉。
- 公开访问模式下，顶部导航只保留内容 tabs。

## 视觉规范

### 信息层级

- 一级层级：内容 tabs。
- 二级层级：系统菜单按钮。
- 展开层：系统抽屉。

这个层级应直接体现在颜色和体积上，而不是只靠文案说明。

### 颜色语义

- 内容导航：蓝色主态，表示主浏览路径。
- 系统导航：橙色辅助态，表示配置/管理类动作。
- 抽屉背景：比正文卡片更轻，避免喧宾夺主。

### 材质

- 顶部导航条使用轻量玻璃纸感或半透明纸感浮层。
- 抽屉和正文之间用细阴影或边框区分，不做厚重投影堆叠。
- 禁止再次出现“整块导航卡片占满顶部”的视觉感受。

## 实现边界

### 主要改动文件

- `src/server/renderAppLayout.ts`
  - 新增移动端顶部导航结构。
  - 保留桌面端侧边栏结构不变。
- `src/server/public/site.css`
  - 新增移动端顶部导航、tabs、系统菜单按钮、抽屉样式。
  - 在移动断点隐藏侧边栏内原有 `nav-group`，避免重复导航。
- `src/server/public/site.js`
  - 新增系统抽屉的开关逻辑。
  - 处理点击外部收起。
  - 处理内容 tabs 点击前的抽屉收起。

### 不修改的行为

- 不修改任何页面路由。
- 不修改统一壳层桌面端菜单结构。
- 不修改系统菜单与内容菜单的路由定义来源。
- 不修改内容页与系统页内部模块。

## 测试要求

### 服务端渲染测试

- 确认移动端顶部导航结构存在。
- 确认内容 tabs 与系统菜单按钮都能由当前路由元数据正确生成。
- 确认公开访问模式下不渲染系统菜单按钮/抽屉结构。

### 前端脚本测试

- 确认系统菜单按钮能切换抽屉展开/收起状态。
- 确认点击外部区域会收起抽屉。
- 确认点击内容 tabs 时会关闭抽屉。

### CSS 资产测试

- 锁定 `sticky` 浮层、横向 tabs、系统抽屉与移动端隐藏原侧栏导航的关键规则。
- 锁定移动端正文顶部补偿，避免导航遮住页面内容。

## 验收标准

### 可用性验收

- 用户在移动端进入内容页时，无需滚动即可看到内容导航 tabs。
- 用户打开系统菜单时，能立刻识别 3 个系统入口并完成跳转。
- 顶部导航不会因为系统入口存在而显著压缩正文首屏。

### 视觉验收

- 移动端顶部的整体观感应更像 app 的顶部导航，而不是缩小版侧边栏。
- `内容菜单` 与 `系统菜单` 的语义差异清楚，但视觉骨架统一。
- 抽屉展开后仍然轻量，不应形成第二个“重型头部”。

### 回归验收

- 桌面端导航结构与行为保持不变。
- 公开访问模式下隐藏系统菜单的行为保持不变。
- 当前 active 态和路由跳转行为保持不变。

## 设计来源与确认记录

- 用户明确指出当前移动端顶部导航“只是实现了功能，但并不好用”。
- 用户确认只希望 `内容菜单 + 系统菜单` 变为移动端置顶导航，品牌块、页面摘要、主题和账号区继续留在正文流中。
- 用户在备选方案中明确选择 `内容 Tabs + 系统抽屉入口`。
