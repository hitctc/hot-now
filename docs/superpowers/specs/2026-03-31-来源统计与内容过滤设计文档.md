# HotNow Source Analytics And Content Filter Design

## 背景

当前 unified shell 已经完成了多源内容浏览、source 启停、手动采集与手动发信的主链路，但在“数据迭代收集”和“内容浏览控制”两个方向上仍有明显缺口：

- `/settings/sources` 目前更像 source 管理页，只能看到启用状态、最近抓取时间和最近抓取状态
- 管理员看不到每个 source 当前总共有多少条内容
- 管理员看不到每个 source 今天发布了多少、今天被系统抓进来多少
- 管理员看不到每个 source 今天到底有多少内容进入了 `热点资讯`、`热门文章`、`最新 AI 消息` 的候选池，以及最终展示了多少
- `/`、`/articles`、`/ai` 三个内容页目前没有 source 维度的浏览过滤能力，用户无法快速只看某几个来源

用户希望补上的不是独立 BI 系统，而是一套更直接的“来源可见性 + 浏览过滤”能力：

- 在 `数据迭代收集` 页上，一眼看清每个 source 当前产出和今天的栏目采纳效果
- 在内容页上，用符合当前 `Editorial Signal Desk` 风格的轻量复选过滤条控制浏览视角

## 目标

### 核心目标

- 把 `/settings/sources` 从“source 开关页”升级为“采集工作台 + 来源概览页”
- 新增每个 source 的基础产出统计：
  - 总条数
  - 今天发布数
  - 今天抓取数
- 新增每个 source 当天在三个内容栏目的采纳统计：
  - `Hot 入池 / 展示`
  - `Articles 入池 / 展示`
  - `AI 入池 / 展示`
- 在 `/`、`/articles`、`/ai` 顶部新增共享的 source 复选过滤条
- 让三个内容页共享一套 source 勾选状态，并持久化到 `localStorage`

### 用户感知目标

- 管理员进入 `/settings/sources` 后，能快速判断：
  - 哪些 source 今天产出多
  - 哪些 source 今天抓取得多
  - 哪些 source 今天真正被各栏目吃进去了
- 普通访问者或管理员进入内容页后，可以快速切换“只看哪些来源”
- 新增统计和过滤不会把内容页做成重后台，也不会破坏当前统一站点风格

### 非目标

本轮明确不做：

- 历史累计采纳报表
- 近 7 天 / 30 天趋势图
- 趋势 sparkline 或 chart
- URL 参数化的 source 过滤
- 服务端保存每个用户的筛选偏好
- 每个内容页单独保存一套 source 选择
- 过滤未启用 source
- 独立的来源分析子系统或离线统计仓库

## 设计结论

### 唯一推荐方案

本轮采用 `轻分析层 + 共享筛选层` 方案：

- `/settings/sources` 直接基于现有 SQLite 内容库与当前 view rules 实时计算当天统计
- `/`、`/articles`、`/ai` 新增同一套前端 source 过滤条
- 不新增历史聚合表，不把本轮需求扩展成独立报表系统

原因：

- 当前问题本质上是“可见性缺失”和“浏览控制缺失”，不是“趋势分析能力不足”
- 直接复用现有内容库和栏目规则，可以让统计口径和实际页面结果保持一致
- 这条路径改动最集中，也最容易保持当前 unified shell 的视觉风格

### 页面分工

- `/settings/sources`
  - 负责“系统视角”的 source 产出和栏目采纳统计
  - 负责 source 启用/停用与采集操作管理
- `/`、`/articles`、`/ai`
  - 负责“访客视角”的内容浏览过滤
  - 不承载来源统计分析

这两类页面职责必须分离，避免统计口径被访客个人勾选污染。

## 页面级设计

### `/settings/sources`：数据迭代收集页

### 页面结构

页面从上到下调整为三层：

1. `操作带`
   - 保留“手动采集”和“发送最新报告”
   - 保持轻量，不承担比较信息
2. `来源总览表`
   - 新增，作为页面主信息区
   - 负责横向比较所有 source
3. `source 卡片区`
   - 保留现有卡片
   - 主要承接单个 source 的管理与详情

### 来源总览表

总览表采用符合当前视觉系统的 `紧凑纸感表`：

- 每个 source 一行
- 默认按系统 source 固定顺序显示
- 小字号、细分割线、紧凑列距
- 钴蓝用于表头与结构层级
- 亮橙只用于异常或重点强调，不承担大面积底色

### 总览表字段

表格字段固定为：

- `来源`
- `总条数`
- `今天发布`
- `今天抓取`
- `Hot 入池 / 展示`
- `Articles 入池 / 展示`
- `AI 入池 / 展示`

后三列都采用双值单格表达，例如 `12 / 4`：

- 左值表示 `入池数`
- 右值表示 `展示数`

这样可以在不牺牲扫读效率的前提下，把核心栏目采纳信息压缩进稳定表格宽度。

### source 卡片区

现有 source 卡片继续保留，但角色改为“详情与管理”，不再承担横向比较。

卡片继续承接：

- source 名称与基础说明
- 当前启用 / 停用状态
- 最近一次抓取时间
- 最近一次抓取状态
- RSS 地址或来源说明
- 一行短摘要统计，例如：
  - `总 248 | 今发 11 | 今抓 9`

总览表负责“比较”，卡片负责“管理”，这条分工不做混合。

## 内容页 source 过滤条

### 作用页面

新增过滤条的页面固定为：

- `/`
- `/articles`
- `/ai`

三个页面共享同一套 source 选择状态。

### 放置位置

过滤条放在每个内容页的 intro 区域下方，默认可见，不做二次展开入口。

它应满足两个视觉要求：

- 足够轻，不抢内容页首屏主内容
- 滚动后保持悬浮，方便边浏览边筛选

### 视觉与密度

过滤条采用 `小号、紧凑、轻量` 的表现：

- 左侧一个短标签，例如 `来源筛选`
- 中部为所有 `已启用 source` 的小号复选项
- 右侧固定提供两个动作：
  - `全选`
  - `全不选`

过滤条不承担统计、说明文案或复杂折叠逻辑，不做重型工具栏。

### 交互行为

过滤条行为固定如下：

- 默认状态为全选
- 三个内容页共享同一套勾选结果
- 状态写入浏览器 `localStorage`
- 未登录访问者同样可用
- 只显示当前系统中 `已启用` 的 source
- `全不选` 合法，不自动回退
- `全不选` 后显示空结果态

### 空结果态

当用户主动取消全部 source 时，页面不报错、不偷偷恢复默认值，而是明确进入空结果态：

- 主文案：`当前未选择任何数据源`
- 辅助动作：提供 `重新全选`
- 保留页面壳层、intro 和过滤条

## 统计口径

### 时间口径

所有“今天”相关统计统一按 `Asia/Shanghai` 计算。

### source 自身产出统计

#### 总条数

定义：当前内容库中属于该 source 的全部内容条数。

用途：回答“这个 source 累积沉淀了多少内容”。

#### 今天发布

定义：该 source 中 `published_at` 落在今天的内容条数。

用途：回答“这个 source 今天本身发了多少内容”。

#### 今天抓取

定义：该 source 中按内容入库时间字段判断，今天被系统写入内容库的条数。

用途：回答“系统今天实际抓到了多少新内容”。

### 栏目采纳统计

栏目采纳统计只看“今天”的内容，不提供累计值。

#### 入池数

定义：该 source 今天的内容里，满足对应栏目候选规则并进入候选集的条数。

对应栏目：

- `Hot`
- `Articles`
- `AI`

入池数必须严格复用当前栏目候选逻辑，不允许另起一套“近似规则”。

#### 展示数

定义：候选集再经过当前栏目排序、limit 和展示规则后，最终出现在该栏目的条数。

这里的 `展示数` 固定按 `系统默认内容页口径` 计算：

- 不受访客 `localStorage` source 勾选影响
- 视为所有已启用 source 默认全选

这样页面上的 source 采纳统计才是稳定的“系统视角”，而不是因人而异的“访客视角”。

## 状态与持久化

### 内容页过滤状态

- 介质：浏览器 `localStorage`
- 范围：`/`、`/articles`、`/ai` 共享
- 默认值：所有已启用 source 全选

状态只用于浏览过滤，不反向写回系统配置。

### 系统配置与浏览过滤的边界

必须明确：

- `/settings/sources` 中的启用/停用是系统级配置
- 内容页顶部过滤条只是访客浏览层过滤器

两者不可混用：

- 过滤条不能修改 source 是否启用
- source 启用/停用也不是访客过滤的历史记录界面

## 边界与限制

### 本轮明确不扩展的方向

为避免实现膨胀，本轮明确不引入：

- 历史采纳统计快照
- 累计展示数 / 累计入池数
- 近 7 天、近 30 天维度切换
- 每栏目单独筛选状态
- 来源过滤与 URL 同步
- 基于用户账号的服务端偏好保存

### 风险点

#### 1. 入池口径必须和现有栏目规则一致

如果实现时对 `hot / articles / ai` 的候选逻辑做了“近似重算”，总览表上的入池数就可能和真实内容页不一致。

#### 2. 展示数必须固定用系统默认口径

如果展示数跟随访客 localStorage 过滤变化，`/settings/sources` 将失去稳定统计意义。

#### 3. 过滤条不能演变成第二个 source 管理器

它只能影响“看什么”，不能影响“系统抓什么”或“source 是否启用”。

## 验收标准

本轮设计完成后，用户应该能够做到：

- 在 `/settings/sources` 一眼比较所有 source 的：
  - 总条数
  - 今天发布数
  - 今天抓取数
  - 今天 `Hot / Articles / AI` 的入池与展示
- 继续在同一页面管理单个 source 的启用状态与查看最近抓取结果
- 在 `/`、`/articles`、`/ai` 顶部用统一过滤条勾选来源
- 刷新页面后保留 source 勾选结果
- 主动 `全不选` 后看到明确空态，而不是被系统悄悄恢复

## 后续实现指引

实现时建议遵循以下顺序：

1. 先补 `/settings/sources` 的统计查询与总览表
2. 再补内容页共享 source 过滤状态和顶部过滤条
3. 最后收紧空态、悬浮样式和细节文案

这样可以先保证“统计口径成立”，再收口浏览体验。
