# HotNow 当前页来源指标对齐设计

## 背景

当前项目里已经同时存在两套来源指标语义：

- 内容页来源胶囊，历史上按“该来源单独选中时会展示多少条”的稳定口径显示
- `/settings/sources` 来源统计概览，近期又先后出现过“默认多源上下文”和“来源单独选中”两种实现口径

这导致用户在同一天、同一批内容下，会看到：

- 内容页卡片流明明展示了很多今日内容
- 但来源统计概览里某些来源却显示 `0`
- 或者来源胶囊里的数字与工作台统计看起来不是一套规则

用户已经明确要求：

- 工作台与内容页的来源相关数字需要完全对齐
- 不再混用“稳定来源结果”和“当前页真实结果”两套上下文
- 如果需要牺牲“稳定性”，优先保证“和用户当前看到的页面结果一致”

## 目标

- 让 `/settings/sources` 的来源统计概览与内容页当前结果完全对齐
- 让内容页来源胶囊与工作台来源统计概览使用同一套来源级统计口径
- 明确“今日候选 / 今日展示 / 今日占比”的定义，避免再出现语义漂移
- 保留现有内容排序与来源筛选交互，不新增额外页面设置

## 非目标

本轮明确不做：

- 不改变内容入库逻辑
- 不改变内容评分规则
- 不改变 `AI 新讯 / AI 热点` 页面卡片排序公式
- 不新增新的统计配置项
- 不保留“来源单独选中时的稳定结果”作为主展示语义

## 核心结论

这轮统一采用 **当前页面真实结果口径**。

含义是：

- 所有来源级数字都以“当前浏览器里正在生效的来源筛选集合”为上下文
- 如果用户勾选了 `Juya`，相关来源数字要反映 `Juya` 已经参与当前页混排后的真实结果
- 如果用户去掉某个来源，该来源不再参与当前页结果，则相关统计也同步变化

排序偏好说明：

- 当前排序项只影响展示顺序，不改变来源级条数本身
- 真正会改变来源级条数的，是当前页面来源筛选集合

## 方案比较

### 方案 A：继续使用“来源单独选中时的稳定结果”

优点：

- 单个来源的数字不会随着当前筛选上下文变化

缺点：

- 与用户当前页面真实看到的结果不一致
- 在多源混排场景里，极易出现“页面明明有内容，但工作台来源显示 0”或相反的错觉
- 已被用户明确否决

### 方案 B：全部切到“当前页面真实结果”，推荐

做法：

- 内容页来源胶囊改为显示当前页面真实来源分布
- `/settings/sources` 来源统计概览改为显示当前页面真实来源分布
- 工作台和内容页共享同一组来源筛选上下文

优点：

- 用户看到的所有来源数字都与当前页面实际结果一致
- 概念最少，解释成本最低
- 与用户目标完全一致

缺点：

- 数字会随着来源筛选变化
- 不能再把它理解成来源的“稳定能力”

### 推荐结论

采用 **方案 B**。

## 数据定义

### 1. 今日候选

定义：

- 在当前页面真实来源筛选集合下
- 某来源今天进入当前 view 候选集的条数

“今天”的判断规则：

- 优先使用 `published_at`
- 若缺失，再回退到 `fetched_at`
- 若仍缺失，再回退到 `created_at`
- 时间边界按 `Asia/Shanghai` 自然日

### 2. 今日展示

定义：

- 在当前页面真实来源筛选集合下
- 某来源今天最终出现在当前页面卡片流里的条数

这里的“最终出现”指的是：

- 已通过当前 view 的候选逻辑
- 已参与当前页面真实多源混排
- 已经过当前页面最终展示截断

### 3. 今日占比

定义：

- `某来源今日展示条数 / 当前页面今日实际展示总条数`

展示格式：

- 一位小数百分比
- 例如 `42.9%`

分母说明：

- 分母是当前页面今天实际展示出来的总条数
- 不是配置上限 `20`
- 不是该来源单独选中时的展示总条数

若当前页面今天实际展示总条数为 `0`：

- 显示 `0.0%`

## 页面设计

### 内容页来源胶囊

当前内容页来源胶囊后面的数字，统一改为：

- 当前页面真实来源筛选集合下
- 该来源在当前页面实际展示的条数

这意味着：

- 不再显示“该来源单独选中时会展示多少条”
- 改成显示“在你当前这页真实结果里，这个来源占了多少条”

### `/settings/sources` 来源统计概览

列顺序调整为：

- `来源`
- `总条数`
- `今天发布`
- `今天抓取`
- `AI 新讯今日候选 / 今日展示`
- `AI 新讯当前页今日占比`
- `AI 热点今日候选 / 今日展示`
- `AI 热点当前页今日占比`

其中：

- `今日候选 / 今日展示` 使用 `a / b` 形式
- `今日占比` 独立成列，不再塞进同一个单元格里

## 上下文同步规则

为了让内容页和工作台真正对齐，这轮需要建立一条明确规则：

- 内容页当前正在使用的来源筛选集合，是来源级统计的唯一上下文来源

具体要求：

- 内容页来源筛选变化后，内容页自身来源胶囊数字即时更新
- `/settings/sources` 读取同一份来源筛选上下文后，来源统计概览也显示相同口径的结果

本轮不要求工作台去复刻内容页视觉，只要求统计语义一致。

## 技术落点

### 1. 内容选择层

在 `buildContentViewSelection` 上补充当前页面真实来源分布统计输出：

- 当前页候选数量按来源分组
- 当前页展示数量按来源分组
- 当前页今日候选数量按来源分组
- 当前页今日展示数量按来源分组
- 当前页今日展示总条数

这些统计必须基于：

- 当前 `selectedSourceKinds`
- 当前页面 view
- 当前 reference time

### 2. 内容页模型

`buildContentPageModel` 返回的 `sourceFilter.options` 需要从单一 `visibleCount` 扩展成更明确的统计字段，例如：

- `todayCandidateCount`
- `todayVisibleCount`
- `todayVisibleShare`

旧的稳定 `visibleCount` 语义在本轮废弃。

### 3. 工作台模型

`listSourceWorkbench` 不再独立推导“默认多源结果”或“来源单独选中结果”。

它需要接收或复用：

- 当前页面真实来源筛选集合

并按该集合返回：

- `ai.todayCandidateCount`
- `ai.todayVisibleCount`
- `ai.todayVisibleShare`
- `hot.todayCandidateCount`
- `hot.todayVisibleCount`
- `hot.todayVisibleShare`

## 接口影响

### 内容页接口

`/api/content/ai-new`
`/api/content/ai-hot`

来源筛选 options 返回字段语义变化：

- 来源胶囊数字不再是“单来源稳定展示条数”
- 改成“当前页真实展示条数”

### 系统页接口

`/api/settings/sources`

需要支持与内容页一致的来源筛选上下文输入，避免再次退回服务端默认口径。

推荐沿用现有 header：

- `x-hot-now-source-filter`

如果本轮实现中发现仅靠 header 还不够，再补一个轻量 query 或 body 方案，但默认先复用现有 header。

## 测试要求

至少补以下验证：

1. 内容页来源胶囊数字
- 在多源上下文下，来源胶囊显示当前页真实展示条数
- 不再显示稳定来源结果

2. 工作台统计概览
- `AI 新讯` 与 `AI 热点` 的“今日候选 / 今日展示”跟当前页真实结果一致
- `今日占比` 为一位小数百分比
- 当前页今天总展示为 `0` 时，占比安全显示 `0.0%`

3. 上下文一致性
- 同一份来源筛选上下文下，内容页与工作台结果一致
- 勾选 `Juya` 前后，两个地方的来源指标同时变化

4. 回归
- 现有来源筛选持久化行为不变
- 排序切换不应错误改变来源条数

## 迁移说明

本轮完成后，需要把旧的“稳定来源结果”设计视为 superseded。

尤其是：

- 内容页来源胶囊
- 来源统计概览

都不再允许继续使用“该来源单独选中时的稳定展示条数”作为主展示语义。
