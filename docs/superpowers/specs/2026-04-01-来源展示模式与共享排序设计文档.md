# HotNow Source Display Mode And Shared Content Sort Design

## 状态

本设计自提交起生效，并取代仅针对 `Juya AI Daily` 的旧方案。

被替代文档：

- `docs/superpowers/specs/2026-04-01-Juya全量Feed可见性设计文档.md`

## 背景

当前内容页已经具备两类能力：

- 按 source 做浏览过滤
- 按统一内容池规则生成栏目可见结果

但这两层能力还不能满足新的产品要求。

### 当前缺口

#### 1. source 过滤仍然只是“先过滤，再截断”

现在即使用户已经在内容页里只看某个来源，该来源的内容仍然会继续受栏目 `limit` 截断。  
这意味着 source filter 只能回答“看哪些来源”，不能回答“这个来源要不要完整展开”。

#### 2. 来源展示策略还没有系统级持久化配置

当前 `content_sources` 只有：

- source 是否启用
- source 基础信息
- 最近采集状态

还缺一个系统级展示策略开关，无法表达：

- “当用户勾选这个来源时，给它完整展开”
- “其他来源仍按原有栏目精选”

#### 3. 内容页排序方式是固定的

用户现在不能在内容页上切换：

- 按发布时间
- 按评分

也没有共享的浏览器本地排序偏好。

## 目标

### 核心目标

- 为每个 source 增加系统级持久化展示开关
- 开关语义是：`当用户在内容页显式勾选该来源时，是否对该来源启用全量展示`
- 新增内容页共享排序选项：
  - `按发布时间`
  - `按评分`
- 排序偏好写入浏览器 `localStorage`，刷新后保持，三个内容页共享
- 新用户首次进入内容页时：
  - 普通来源默认选中
  - 开启“全量展示”的来源默认不选中
- 旧用户如果本地已有筛选记录，系统不覆盖

### 用户感知目标

- 管理员可以在 `/settings/sources` 精确配置“哪些来源被勾选时应完整展开”
- 用户第一次进入内容页时，不会因为全量来源过多而直接看到过长列表
- 用户一旦手动勾选某个“全量展示”来源，就能看到该来源完整内容
- 用户可以在不同内容页之间保持一致的排序体验

### 非目标

本轮明确不做：

- 为 source 建立多档展示模式，而不只是一个布尔开关
- 为不同内容页分别保存不同排序偏好
- 用 URL query 参数持久化排序或 source 过滤
- 修改 `contentScore` 的计算公式
- 修改 `viewRuleConfig` 的字段结构
- 引入来源专属详情页或来源专属内容页
- 修改 `/settings/sources` 的来源统计口径定义

## 设计结论

### 唯一推荐方案

本轮采用 `source 级展示模式 + 内容页共享排序偏好` 方案：

- 在 `content_sources` 表上新增 `show_all_when_selected`
- `/settings/sources` 负责管理这个开关
- 内容选择层在 `visibleCards` 阶段读取该开关
- 内容页在浏览器本地保存一套共享排序偏好

推荐原因：

- 语义和当前系统的 source 管理模型一致
- 只增加一个布尔字段，不需要额外建新表
- 能精确表达“系统级策略 + 用户侧显式勾选才生效”
- 排序偏好和 source 展示策略是正交关系，可以独立组合

## 数据模型

### `content_sources` 新增字段

新增字段：

- `show_all_when_selected INTEGER NOT NULL DEFAULT 0`

语义：

- `0`：该来源仍按原有栏目规则截断
- `1`：当用户在内容页显式勾选该来源时，对该来源启用全量展示

这个字段与 `is_enabled` 的关系必须保持清晰：

- `is_enabled`
  - 控制系统是否采集该来源
  - 控制该来源是否出现在内容页筛选选项中
- `show_all_when_selected`
  - 只控制展示策略
  - 不影响采集启停

### 默认值

所有现有 source 和后续新 seed 的默认值都为：

- `show_all_when_selected = 0`

这样升级后系统行为默认不变，管理员需要显式打开该开关。

## 系统菜单：`/settings/sources`

### 页面职责

`/settings/sources` 继续是 source 的系统级配置页，本轮在现有基础上新增“展示策略”管理能力。

### 每个 source 卡片新增开关

每个来源卡片新增一个独立开关，建议文案固定为：

- `选中该来源时全量展示`

该开关应与“启用 / 停用 source”分开呈现，不混入同一个按钮。

原因：

- `启用 / 停用` 是采集配置
- `全量展示` 是内容浏览配置
- 这两类动作属于不同维度，混在一个交互里会让语义变模糊

### 设置页读取模型

以下读取模型都需要带出新字段：

- `listSources`
- `listSourceCards`
- `/api/settings/sources`
- 如当前 `/settings/sources` 还依赖 `listSourceWorkbench`，则该模型也应带出该字段

## 系统动作接口

### 不复用现有启停接口

不建议复用现有 source 启停接口来顺带改展示模式。

推荐新增独立动作：

- `POST /actions/sources/display-mode`

请求体：

- `kind`
- `showAllWhenSelected`

响应语义：

- 成功返回更新后的 source 展示模式状态
- 未找到 source 返回 `404`
- 未登录返回 `401`
- payload 非法返回 `400`

这样可以保持：

- 采集配置动作
- 展示配置动作

各自独立、便于测试、也便于后续继续扩展。

## 内容页默认 source 勾选

### 旧用户

如果浏览器已有本地 source 筛选记录：

- 完全保留旧值
- 不因系统开关变化而自动移除或覆盖任何来源

这是本轮明确确认的规则。

### 新用户或无本地记录用户

如果浏览器还没有本地筛选记录：

- 默认选中所有 `is_enabled = true` 且 `show_all_when_selected = false` 的来源
- 默认不选中所有 `is_enabled = true` 且 `show_all_when_selected = true` 的来源

这样可以避免用户首次进入内容页时，被可能非常长的“全量来源”列表直接打满。

### 勾选触发条件

来源开关只有在下面两个条件同时满足时才生效：

1. 当前来源的 `show_all_when_selected = true`
2. 当前用户在内容页里显式勾选了这个来源

只满足其中一个条件都不生效。

## 内容页排序偏好

### 排序选项

新增共享排序方式：

- `published_at`
- `content_score`

用户可见文案：

- `按发布时间`
- `按评分`

### 默认值

默认排序固定为：

- `按发布时间`

### 持久化

排序偏好写入浏览器 `localStorage`：

- 刷新后保持
- `/`
- `/articles`
- `/ai`

三页共享同一套偏好

### 本轮不做

- 不对不同页面分别保存不同排序设置
- 不把排序状态写回服务端
- 不把排序状态加入系统菜单配置

## 内容选择逻辑

### 维持现有两层模型

继续沿用：

- `candidateCards`
- `visibleCards`

不修改：

- `contentScore` 计算公式
- `viewRuleConfig` 字段定义
- source priority 语义

### `candidateCards`

`candidateCards` 仍然按当前逻辑生成：

1. 按 source filter 过滤
2. 读取 view rule
3. 计算 ranking score
4. 得到完整候选集

这一层不引入“全量来源”特例。

### `visibleCards`

`visibleCards` 的生成逻辑改为：

1. 先从 `candidateCards` 中识别出 `fullDisplaySourceKinds`
2. `fullDisplaySourceKinds` 的判断条件：
   - source 在当前用户筛选中被显式选中
   - 对应 `content_sources.show_all_when_selected = true`
3. 把候选集拆成两组：
   - `fullDisplayCards`
   - `limitedCards`
4. `fullDisplayCards` 全部进入 `visibleCards`
5. `limitedCards` 继续按当前栏目 `limit` 截断
6. 合并后形成最终 `visibleCards`

因此，新语义是：

- 开启全量展示且被显式勾选的来源：免 `limit`
- 其他来源：继续受 `limit` 约束

### 为什么不直接把 `limit` 调大

不采用“勾中某些来源时把整个页面 limit 调大”的方案，原因如下：

- 会放大所有来源，而不是只放大命中的来源
- 会让 `limit` 从“栏目精选上限”变成“动态来源特例开关”
- 不能清晰表达“某些来源全量、某些来源精选”的混合语义

## 最终排序逻辑

### 排序发生时机

排序应发生在最终 `visibleCards` 集合上，而不是在“是否入选”之前。

原因：

- “全量展示”解决的是入选问题
- “按发布时间 / 按评分”解决的是显示顺序问题

两者语义不能混在同一步里处理。

### `按发布时间`

规则：

- 首先按 `published_at` 新到旧
- 缺失或非法时间时，回退到当前使用的 ranking timestamp
- 再相同则按 `id` 或现有稳定 tie-break 规则

### `按评分`

规则：

- 首先按 `contentScore` 高到低
- 分数相同再按发布时间新到旧
- 再相同则按当前稳定 tie-break 规则

### 混合列表语义

最终列表保持为一条统一混排流：

- 不拆成“全量来源区”和“普通来源区”
- 不强行把全量来源固定排在最前
- 只按用户当前排序方式决定显示顺序

这条设计是已明确确认的产品结论。

## 页面表现

### 只勾选全量来源

如果用户只勾选了一个开启“全量展示”的来源：

- 页面显示该来源全部内容
- 再按当前排序方式输出

### 勾选全量来源 + 普通来源

如果用户同时勾选了：

- 一个或多个全量来源
- 一个或多个普通来源

则结果为：

- 全量来源完整进入可见列表
- 普通来源继续按栏目 `limit` 精选
- 最终按“发布时间”或“评分”统一混排

### 不勾选任何全量来源

如果当前筛选里没有命中任何开启“全量展示”的来源：

- 行为完全保持现状

## 与 source analytics 的关系

`/settings/sources` 中现有 source analytics 使用：

- `candidateCards`
- `visibleCards`

来计算栏目“入池 / 展示”。

本轮引入全量展示后：

- 某个来源如果同时满足“开启全量展示 + 被用户显式勾选”的用户侧条件，
  不会改变系统工作台的统计口径
- 系统工作台仍然代表“系统默认视角”

因此，本轮推荐的统计口径是：

- source analytics 不读取用户本地 source filter
- 只使用系统默认内容页视角
- 排序偏好也不影响 analytics

换句话说：

- source analytics 保持系统视角
- 全量展示和排序偏好只影响用户内容页视角

## 客户端状态与存储

### source filter

继续沿用现有 source filter `localStorage` 存储，但初始化逻辑更新为：

- 若本地已有值：直接使用
- 若本地无值：按新的默认选中规则初始化

### sort preference

新增一个独立 `localStorage` key，例如：

- `hot-now-content-sort`

值域：

- `published_at`
- `content_score`

无效值回退到：

- `published_at`

### 共享范围

以下页面共享：

- `/`
- `/articles`
- `/ai`

## 风险与取舍

### 主要风险

- 列表长度可能显著增加
- “全量来源 + 普通来源精选”的混合结果会让列表语义更复杂
- 用户如果开启多个全量来源并同时勾选，页面长度会进一步增加

### 为什么接受这些风险

这些后果正是产品希望换来的能力：

- 对特定来源做完整可见
- 但不把其他来源也一起放大
- 同时让排序权交给用户

## 交付边界

本轮交付完成的标志是：

- 每个 source 都可以在系统里配置“选中时全量展示”
- 新用户默认不会自动选中这些全量来源
- 旧用户的本地筛选习惯不被覆盖
- 内容页支持“按发布时间 / 按评分”切换
- 排序偏好本地持久化并在三个内容页共享
- 内容页仍保持统一混排流，不拆专区
