# HotNow AI Content Search Design

## 1. 背景

当前 unified shell 的内容页已经收口为两条主线：

- `AI 新讯`
  - 路由：`/` 与 `/ai-new`
  - API：`/api/content/ai-new`
- `AI 热点`
  - 路由：`/ai-hot`
  - API：`/api/content/ai-hot`

这两页现在已经具备：

- 共享 source 复选过滤
- 共享排序切换
- 固定 `50` 条 / 页的分页

但仍缺一个很直接的浏览能力：

- 用户无法在 `AI 新讯 / AI 热点` 中快速按标题查找特定内容

用户这次要的不是全站搜索，也不是复杂检索系统，而是一套很轻的内容页标题搜索：

- 仅作用于 `AI 新讯 / AI 热点`
- 只匹配标题
- 通过 `回车 / 点按钮` 才触发
- 搜索词不写进 URL
- 输入框内带可点击的清空 icon
- 搜索词在 `AI 新讯 / AI 热点` 两页共用
- 搜索词写入 `localStorage`

同时，用户明确接受：

- 搜索应发生在完整结果集上，而不是仅搜当前页
- 搜索完成后再分页，避免漏掉后续页里的匹配结果

## 2. 目标

本轮设计目标只有 4 个：

1. 为 `AI 新讯 / AI 热点` 增加一套共享的标题搜索能力。
2. 保持搜索交互轻量，不引入“边输入边请求”的实时搜索成本。
3. 保持结果完整性，确保搜索命中是在整套结果集上完成，再分页展示。
4. 让搜索状态在 `AI 新讯 / AI 热点` 两页间共享，并在刷新后继续保留。

## 3. 非目标

本轮不做以下事情：

- 不做摘要、正文或来源名搜索
- 不做全文检索、SQLite FTS 或额外搜索索引
- 不做 URL query 搜索参数
- 不做浏览器前进 / 后退回放搜索历史
- 不做高亮命中词
- 不做联想词、搜索建议或历史搜索记录
- 不做系统页、legacy 页或其他页面的搜索能力
- 不做“输入中实时搜索”或防抖请求

## 4. 设计结论

### 4.1 唯一推荐方案

采用“顶部紧凑搜索框 + 共享本地关键词 + 标题过滤后分页”的方案：

- 页面顶部筛选带
  - 增加一枚紧凑搜索框
- 搜索范围
  - 仅匹配标题
- 触发方式
  - `Enter` 或点击 `搜索`
- 清空方式
  - 输入框内部提供可点击清空 icon
- 搜索状态
  - `AI 新讯 / AI 热点` 共用
  - 写入 `localStorage`
  - 不写进 URL
- 结果构建
  - 先得到完整结果集
  - 再按标题关键词过滤
  - 最后分页

### 4.2 为什么不采用“当前页内搜索”

如果只在当前页 50 条里搜索：

- 会漏掉后续页里的真实命中内容
- 用户会误以为“系统搜不到”
- 搜索和分页的关系会变得不可信

因此本轮明确固定为：

- `完整结果集搜索`
- `搜索后分页`

## 5. 页面与交互设计

### 5.1 搜索框位置

搜索框放在内容页现有顶部筛选带中，和 source 过滤、排序切换处于同一层级。

布局收口为：

- 左侧：source 过滤
- 中间：排序切换
- 右侧：标题搜索

这样可以保持语义清楚：

- source 过滤控制“看哪些来源”
- 排序控制“怎么排”
- 搜索控制“找什么标题”

搜索框需要保持：

- 紧凑
- 小尺寸
- 不抢主内容
- 跟随顶部筛选带一起在滚动时悬浮

### 5.2 搜索框长相

搜索框采用轻量输入框样式，不做大块工具条式搜索。

最低交互元素包括：

- 左侧搜索 icon
- 中间输入区域
- 输入框内右侧清空 icon
- 右侧一个小号 `搜索` 按钮

其中：

- 输入框内清空 icon 仅在当前输入值非空时可见
- 清空 icon 点击后立即清空当前输入并恢复默认结果

### 5.3 触发方式

本轮搜索只允许两种触发方式：

- 输入后按 `Enter`
- 点击 `搜索` 按钮

不做：

- 边输入边实时搜索
- 输入中自动请求

这样可以减少不必要请求，也更符合当前页面“浏览控制器”而不是“搜索产品”的定位。

## 6. 搜索状态模型

### 6.1 状态拆分

搜索状态必须分成两层：

1. `当前输入值`
   - 用户正在编辑的内容
2. `当前生效关键词`
   - 真正参与结果过滤的关键词

这样可以避免：

- 输入了一半就刷新结果
- 尚未确认的半成品关键词污染内容列表

### 6.2 共享范围

搜索词只在下面两页共享：

- `AI 新讯`
- `AI 热点`

它不外溢到：

- `/settings/*`
- legacy 页面
- 其他未来页面

### 6.3 本地持久化

当前生效关键词写入浏览器本地 `localStorage`。

建议 key 收口为：

- `hot-now-content-search`

页面首次进入时：

- 如果本地存在该值，则用它初始化当前生效关键词和输入框默认值
- 如果为空或不存在，则按“未搜索”状态进入

### 6.4 URL 约束

搜索词不写进 URL。

因此：

- `/ai-new?page=2` 仍然只表达分页状态
- 搜索关键词只留在页面状态和 `localStorage`

这轮明确不做：

- `?q=...`

## 7. 结果构建顺序

### 7.1 唯一顺序

本轮内容页结果构建顺序固定为：

1. source 过滤
2. 排序
3. 标题搜索
4. 分页

也就是：

- 先得到当前页面本来应该展示的完整结果集
- 再对这套结果按标题关键词过滤
- 最后按 `50` 条 / 页分页

### 7.2 标题匹配语义

标题搜索采用轻量字符串匹配语义：

- 先对关键词做 `trim`
- 纯空白视为“无搜索”
- 标题匹配按大小写不敏感处理
- 匹配规则采用“包含即可”

本轮不做：

- 分词
- 多关键词语法
- 布尔表达式
- 高级搜索语法

### 7.3 提交与清空时的分页行为

下面两种动作都必须自动回到第一页：

- 执行一次新搜索
- 清空搜索词

原因很直接：

- 搜索后结果集可能变短
- 旧页码可能已经失效
- 回到第一页最稳定，也最符合用户预期

## 8. 空结果态

### 8.1 搜索为空时

当搜索关键词生效后，没有任何标题命中时，页面应显示搜索专属空态，而不是复用默认内容空态。

统一文案收口为：

- 标题：`没有找到匹配的内容`
- 描述：`可以换个关键词，或清空搜索后查看全部结果。`

### 8.2 空态优先级

空态优先级按下面顺序判断：

1. 内容库损坏等 degraded 状态
2. `全不选` source 导致的空结果
3. 搜索命中为空
4. 页面原本的默认空态

这样可以避免把“系统问题”“过滤为空”“搜索为空”混成一种提示。

## 9. API 与前后端边界

### 9.1 传输方式

因为搜索词不进 URL，本轮建议沿用当前内容页已有 header 风格，把搜索词通过请求头传给内容 API。

建议 header 名称收口为：

- `x-hot-now-content-search`

这样和现有这两类 header 保持一致：

- `x-hot-now-source-filter`
- `x-hot-now-content-sort`

### 9.2 页面与服务端职责

职责拆分如下：

- 页面层
  - 管理输入值
  - 管理当前生效关键词
  - 管理 `localStorage`
  - 在提交、清空、翻页时触发加载
- 内容页模型 / 内容 API
  - 接收搜索词
  - 在完整结果集上执行标题过滤
  - 再返回分页后的卡片与分页元数据

本轮不建议做成：

- 纯前端当前页内过滤

因为那样会直接破坏搜索结果完整性。

## 10. 性能与边界

### 10.1 当前性能策略

本轮明确采用轻量方案：

- 不上 SQLite FTS
- 不做额外索引
- 不做实时搜索
- 只做标题包含匹配

在当前项目规模下，这是可接受的，因为：

- `AI 新讯` 已经有 `24 小时` 窗口
- `AI 热点` 当前数据规模仍适合轻量过滤
- 标题匹配只是对完整结果集做一层字符串判断，成本远低于前面的规则构建与排序

### 10.2 后续升级路径

如果未来数据量明显增长，再按下面顺序升级：

1. 先把标题过滤下推到 SQLite `LIKE`
2. 再评估是否需要 SQLite FTS

本轮不提前做这两步。

## 11. 实施边界

本轮只影响：

- `AI 新讯`
- `AI 热点`
- 内容页共享搜索状态
- 内容页 API / 模型层的标题过滤

本轮不改：

- `/settings/*`
- legacy 内容页
- source 工作台统计口径
- 现有分页大小
- source 过滤与排序的已确认语义

## 12. 验收标准

满足以下条件即视为本轮达标：

1. `AI 新讯 / AI 热点` 顶部都能看到紧凑搜索框。
2. 输入关键词后，只有 `Enter / 点击搜索` 才触发结果刷新。
3. 搜索只匹配标题，不影响 source 过滤和排序口径。
4. 搜索发生在完整结果集上，再分页。
5. 搜索或清空后都会回到第一页。
6. 清空 icon 可点击，并能恢复默认结果。
7. 搜索词在 `AI 新讯 / AI 热点` 两页共享，并在刷新后保留。
8. 搜索无命中时，页面显示专属空态，而不是默认空态或报错态。
