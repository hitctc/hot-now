# HotNow AI Content Strategy And Pagination Design

## 1. 背景

当前内容页已经收口为两条主线：

- `AI 新讯`
  - 路由：`/` 与 `/ai-new`
  - API：`/api/content/ai-new`
- `AI 热点`
  - 路由：`/ai-hot`
  - API：`/api/content/ai-hot`

但这两页当前仍有两个明显问题：

1. `/settings/view-rules` 和内容页真实生效逻辑没有真正接通  
   当前内容页仍然直接读取内部默认策略，而不是把设置页确认过的规则作为真实来源。

2. `AI 新讯` 与 `AI 热点` 的展示上限仍受旧的“截前 20 条”语义影响  
   这会把实际符合条件的内容直接砍掉，既不适合“新讯”场景，也不适合“热点”场景。

用户这次要的不是微调排序，而是明确重写这两页的结果边界：

- `AI 新讯`
  - 严格只看 `24 小时内`
  - 只要符合规则都进入结果集
  - 通过分页展示，不再总量截断
- `AI 热点`
  - 不能强压成 `24 小时`
  - 只要符合热点逻辑都进入结果集
  - 同样通过分页展示，不再总量截断

同时，用户明确要求：

- `/settings/view-rules` 必须和内容页真实生效逻辑接起来
- `AI 新讯` 与 `AI 热点` 都支持分页
- 每页固定 `50` 条

## 2. 目标

本轮设计目标只有 4 个：

1. 让 `/settings/view-rules` 成为 `AI 新讯 / AI 热点` 的真实策略来源。
2. 把 `AI 新讯` 收口成“严格 24 小时窗口内的新信号流”。
3. 把 `AI 热点` 收口成“按热点规则筛出的长期热度池”，不做人为 24 小时压缩。
4. 把两页的旧“总量截断”改成“分页展示”，每页 `50` 条。

## 3. 非目标

本轮不做以下事情：

- 不新增第三类内容页
- 不改 source 勾选过滤的交互口径
- 不改收藏、反馈、点赞、点踩链路
- 不新增无限滚动
- 不做复杂的时间范围切换器
- 不把分页大小做成用户可配置项
- 不做“全部内容页统一分页策略”之外的额外列表系统重构

## 4. 设计结论

### 4.1 唯一推荐方案

采用“策略页接线 + 页面语义分流 + 固定分页”的方案：

- `/settings/view-rules`
  - 继续承担规则配置入口
- 内容页 API
  - 改为读取设置页保存后的真实规则
- `AI 新讯`
  - 增加硬性的 `24 小时内` 时间过滤
- `AI 热点`
  - 保持热点规则主导，不加 24 小时硬窗
- 两页结果集
  - 都不再用旧的“前 20 条”总量截断
  - 统一改为分页切片，每页 `50` 条

### 4.2 为什么不再保留“总量 limit 截断”

对 `AI 新讯` 而言，核心问题是“时间窗口内的新内容尽量完整地看见”，不是“只保留前 20 条”。

对 `AI 热点` 而言，核心问题是“热点形成逻辑本身”，不是“先砍掉大部分结果再说”。

因此，这两页的结果控制方式都应从：

- `策略决定最多展示多少条`

改成：

- `策略决定什么内容进入结果集`
- `分页决定用户每次看到哪 50 条`

## 5. 页面语义

### 5.1 AI 新讯

`AI 新讯` 的职责收口为：

- 展示最近 `24 小时` 内新进入窗口的 AI 内容
- 更偏“新消息、模型更新、产品动态、事件信号”
- 结果集完整保留，不因旧 limit 被硬截断

### AI 新讯过滤规则

`AI 新讯` 的结果必须同时满足两层条件：

1. 满足 `AI 新讯` 对应的策略规则
2. 时间戳落在最近 `24 小时`

时间戳定义采用下面顺序：

- 优先 `published_at`
- 如果缺失，则回退 `fetched_at`
- 再缺失，则回退 `created_at`

这样可以避免因为部分 source 没有稳定发布时间而被整页漏掉。

这里的 `24 小时` 是硬过滤，不是“新鲜度加分”。

也就是说：

- 超出 24 小时的内容，不进入 `AI 新讯` 结果集
- 不是分数低一点，而是直接不展示

### 5.2 AI 热点

`AI 热点` 的职责收口为：

- 展示已经开始形成热度、值得继续追踪的 AI 内容
- 更偏“持续讨论、传播性、连续影响”
- 不人为压成 24 小时窗口

### AI 热点过滤规则

`AI 热点` 只受两类条件影响：

1. 满足 `AI 热点` 对应的策略规则
2. 满足热点形成逻辑本身

它不再额外叠加：

- `最近 24 小时内`

这样的硬时间门槛。

原因很明确：

- 某些真正重要的热点，本来就会持续几天甚至更久
- 热点页核心是“形成热度并值得继续看”，不是“是否刚刚发生”

## 6. /settings/view-rules 与真实内容页接线

本轮必须明确一条规则：

`/settings/view-rules` 不再只是“配置页”，而是 `AI 新讯 / AI 热点` 的真实规则来源。

这意味着：

- 设置页保存什么规则
- 内容页 API 就按照什么规则构建结果集

不再允许：

- 设置页维护一套规则
- 内容页再偷偷读另一套内部默认值

### 接线原则

规则职责按页面语义分开：

- `AI 新讯`
  - 读取 `AI 新讯` 对应规则
- `AI 热点`
  - 读取 `AI 热点` 对应规则

页面之间不能再共用一套模糊的“AI 默认规则”。

### 规则与分页的边界

这次要明确拆开两件事：

- `view-rules`
  - 决定什么内容进入结果集
- `pagination`
  - 决定结果集如何按页显示

分页不是策略的一部分。

因此本轮不建议把：

- `每页 50 条`

写进 `/settings/view-rules` 作为可编辑字段。

它应是页面层固定契约，而不是策略层配置项。

## 7. 分页设计

### 7.1 分页适用范围

分页只加到：

- `AI 新讯`
- `AI 热点`

### 7.2 分页大小

固定为：

- `50 条 / 页`

本轮不做用户可配置页大小。

### 7.3 分页行为

分页基于“完整结果集”再切片：

1. 先按页面规则拿到完整结果集
2. 再按当前排序规则排序
3. 最后按页码切 `50` 条

这样可以确保：

- 结果集完整性来自规则
- 页内呈现来自排序
- 单页数量来自分页

而不是把这三个概念混成一个旧的 `limit`。

### 7.4 分页状态

分页状态建议使用 URL query 里的 `page`，例如：

- `/ai-new?page=2`
- `/ai-hot?page=3`

`page` 采用 `1-based` 语义：

- `page=1` 表示第一页
- 缺失、非数字、`<= 0` 时统一回退到 `1`

原因：

- 翻页是导航状态，不是偏好设置
- 用户刷新后应停留在当前页
- 前进后退和分享链接也更自然

本轮不建议把分页状态写进 `localStorage`。

## 8. 排序与分页关系

两页当前已有排序控制能力，这轮不取消。

排序和分页的关系固定为：

1. 先构建结果集
2. 再排序
3. 再分页

不能反过来写成：

- 先分页
- 再页内排序

否则翻页结果会不稳定。

## 9. 空态与边界态

### 9.1 AI 新讯空态

当 `AI 新讯` 在最近 24 小时内没有符合规则的内容时，空态文案应该强调“当前窗口内暂无结果”，而不是泛泛写成“暂无内容”。

表达重点应是：

- 规则正常
- 页面正常
- 只是当前 `24 小时窗口内` 没有结果

### 9.2 AI 热点空态

当 `AI 热点` 没有命中内容时，空态应强调“当前没有形成热度的内容”，而不是暗示系统异常。

### 9.3 分页越界

如果用户访问超出总页数的 `page`：

- 不报错
- 不返回 500
- 回退到最后一页或返回一个稳定空页，二选一即可

本轮推荐：

- 回退到最后一页

因为这更符合内容浏览场景。

## 10. API 与页面模型调整方向

当前 `ContentPageModel` 需要从“只返回当前页卡片”升级为“返回分页信息 + 当前页卡片”。

建议页面模型增加分页元信息，例如：

- `page`
- `pageSize`
- `totalResults`
- `totalPages`

这样前端才能稳定渲染分页条，而不是靠结果条数猜测。

### AI 新讯 API

`/api/content/ai-new`

应支持：

- 读取 source 过滤
- 读取排序方式
- 读取 `page`
- 返回 `24 小时内` 且满足规则的分页结果

### AI 热点 API

`/api/content/ai-hot`

应支持：

- 读取 source 过滤
- 读取排序方式
- 读取 `page`
- 返回满足热点规则的分页结果

## 11. 风险点

### 11.1 不能再混用内部默认策略和设置页规则

如果这轮只改了分页和时间窗口，但内容页仍然偷读内部默认值，那用户在 `/settings/view-rules` 里改规则依然不会真实生效，这次需求就没有真正完成。

### 11.2 24 小时过滤必须是硬过滤

如果只是把 `freshnessWeight` 调高，而不是做硬时间门槛，那么旧内容仍可能排进 `AI 新讯`，这和用户要求不一致。

### 11.3 热点页不能被新讯逻辑牵着走

`AI 热点` 和 `AI 新讯` 必须是两个独立结果集，不能因为共用一套旧逻辑而被同一时间窗口束缚。

### 11.4 分页必须基于完整结果集

如果仍在分页之前先按旧 limit 截断，分页就只是“对 20 条做分页”，没有意义。

## 12. 验收标准

本轮完成后，应满足以下结果：

- `/settings/view-rules` 改动后，`AI 新讯 / AI 热点` 的真实内容结果会随之变化
- `AI 新讯` 只展示最近 `24 小时` 内且符合规则的内容
- `AI 热点` 不受 `24 小时` 硬窗口限制
- 两页都不再被旧的“前 20 条”总量截断
- 两页都支持分页
- 每页固定 `50` 条
- 分页与排序、source 过滤可以共存

## 13. 后续实现指引

建议实现顺序如下：

1. 先收口内容页真实使用的规则来源，彻底移除“内部默认替代真实配置”的路径
2. 再把 `AI 新讯` 的 `24 小时` 硬过滤落地
3. 再把 `AI 新讯 / AI 热点` 的结果上限改成分页切片
4. 最后补前端分页 UI、空态和测试

这样可以先确保“规则正确”，再补“结果可翻页”。
