# HotNow Content Toolbar Unification Design

## 1. 背景

当前 unified shell 的内容页：

- `/`
- `/ai-new`
- `/ai-hot`

已经提供三类浏览控制能力：

- 共享来源筛选
- 共享排序方式
- 共享标题搜索

但当前顶部控制区仍然按三段结构展开：

1. 一整块来源筛选
2. 一整块排序方式
3. 一整块标题搜索

这会带来两个问题：

- 顶部控制区占用垂直空间过大，压缩了真正的内容首屏
- 三块控件虽然语义相关，但视觉上仍像并列堆叠，而不是一套统一的“内容浏览控制器”

用户本轮要求不是改动筛选、排序或搜索的行为语义，而是把它们整合成一个更紧凑的控制块，减少高度占用。

## 2. 目标

本轮设计目标只有 4 个：

1. 把来源筛选、排序方式和标题搜索整合成一张统一控制卡。
2. 明显减少顶部控制区默认占用的垂直空间。
3. 保留现有 source / sort / search 的行为语义，不引入新的复杂操作。
4. 在桌面端和移动端都保持统一容器语义，而不是重新拆成多个独立模块。

## 3. 非目标

本轮不做以下事情：

- 不改来源筛选的勾选规则
- 不改排序方式的选项与生效逻辑
- 不改标题搜索的范围、触发方式或持久化方式
- 不新增高级筛选、更多维度筛选或 popover 工具菜单
- 不把来源筛选改成抽屉、弹层或独立页面
- 不记忆“来源区展开 / 收起”状态
- 不改内容页分页、空态或内容卡片结构

## 4. 设计结论

### 4.1 唯一推荐方案

采用 `Unified Control Card`：

- 顶部只保留一张 sticky 统一控制卡
- 常显区合并展示：
  - 来源摘要
  - 排序方式
  - 搜索框
- 来源复选项默认折叠
- 点击来源摘要或单独按钮后，在同一张卡片内向下展开来源区

### 4.2 为什么不保留三块并列

即使把现有三块单独缩小：

- 视觉上仍然是三张卡片并排 / 叠排
- 控制区仍会像“设置面板”，而不是“浏览控制条”
- 在移动端依旧会形成重复边框和重复上下间距

因此本轮明确采用：

- 一个容器
- 两层结构
- 按需展开来源区

## 5. 统一控制卡设计

### 5.1 默认常显状态

统一控制卡默认只展开主控制行。

主控制行分成 3 个区域：

- 左侧：来源摘要
- 中间：排序方式
- 右侧：标题搜索

这三个区域都属于同一张卡片，不再各自拥有独立的外层 panel。

### 5.2 来源摘要

来源摘要使用 `数字 + 名称摘要` 形式。

展示规则固定为：

- 选中 `0` 个：
  - `来源：未选择`
- 选中 `1` 个：
  - `来源：OpenAI`
- 选中 `2` 个：
  - `来源：OpenAI、IT之家`
- 选中 `3` 个及以上：
  - `来源：OpenAI、IT之家 +1`

更长列表继续按同样规则处理：

- 始终优先展示前两个来源名
- 其余来源折叠为 `+N`

来源摘要需要满足：

- 摘要文字本身可点击
- 右侧提供单独展开 / 收起按钮
- 两者都能触发展开 / 收起

### 5.3 排序方式

排序方式保留现有两个动作：

- `按发布时间`
- `按评分`

排序依然即时生效，并继续遵循当前行为：

- 切换排序后回到第一页

本轮只收口视觉位置，不改变排序逻辑。

### 5.4 标题搜索

搜索仍然保留当前交互：

- 只搜索标题
- `Enter` 或点击 `搜索` 才触发
- 输入框内保留清空 icon
- 搜索或清空后回到第一页
- 搜索词继续写入 `localStorage['hot-now-content-search']`

搜索区域需要比来源摘要更轻，避免主控制行的视觉重心被搜索框抢走。

## 6. 来源展开区

### 6.1 展开方式

来源筛选区不使用 popover、抽屉或独立浮层。

它必须：

- 在同一张统一控制卡内部向下展开
- 成为控制卡的第二层
- 与主控制行共享背景、边框和整体层级

### 6.2 展开后的内容

展开层保留现有来源筛选能力：

- 所有来源复选项
- `全选`
- `全不选`

来源复选项仍然是多选，不改语义。

### 6.3 展开后的收起规则

本轮固定为：

- 勾选或取消单个来源后，展开层保持打开
- 点击 `全选 / 全不选` 后也保持打开
- 只有用户显式点击摘要或按钮时才收起

这样可以保证连续多选时不会反复展开。

## 7. 响应式布局

### 7.1 桌面端

桌面端保持一张卡，但内部按“主行 + 次行”工作。

主行的宽度优先级为：

1. 来源摘要
2. 排序方式
3. 搜索

也就是：

- 来源摘要占最大可读空间
- 排序占固定紧凑宽度
- 搜索保留稳定输入宽度

展开后第二行出现来源复选项，允许横向流动并在需要时轻换行。

### 7.2 移动端

移动端不强行维持单行布局，但仍然必须保持“一张卡”的容器语义。

移动端布局收口为：

1. 第一行：来源摘要 + 展开按钮
2. 第二行：排序方式
3. 第三行：搜索框

展开来源后：

- 来源区显示在卡片底部
- 复选项按窄屏自然换行
- 仍然保留同一张卡片背景与边框

### 7.3 统一原则

无论桌面端还是移动端：

- 视觉上都只能看到一个控制容器
- 不能退回成三张独立控制块

## 8. 交互细节

### 8.1 Sticky 行为

统一控制卡继续 sticky。

要求保持：

- 默认状态 sticky
- 来源区展开后仍然 sticky
- 滚动时不自动收起展开层

### 8.2 动画

动画只需要服务于“控制区状态清晰”，不应抢内容。

允许的动效只有：

- 来源区展开 / 收起的短高度过渡
- 展开指示符的轻微旋转
- sticky 状态下极轻的层级 / 阴影变化

不做：

- 强浮动感
- 明显位移动画
- 大面积视觉强调

### 8.3 空选提示

如果用户 `全不选`：

- 主控制行中的来源摘要应显示 `来源：未选择`
- 内容区继续沿用现有“未选择任何数据源”的空结果语义

## 9. 实现边界

本轮只改结构整合，不扩行为范围。

### 9.1 会改的部分

- 内容页顶部控制区的容器结构
- 来源摘要与展开按钮
- 来源区折叠 / 展开层
- 桌面端与移动端的统一控制卡布局

### 9.2 不会改的部分

- 来源筛选的业务规则
- 排序选项和排序语义
- 标题搜索语义
- 分页逻辑
- 内容页结果集构建顺序
- 反馈面板和内容卡片行为

## 10. 验证要求

本轮实现完成后，至少验证以下几点：

1. `/`、`/ai-new`、`/ai-hot` 顶部默认只显示一个统一控制卡
2. 来源摘要和右侧按钮都能展开 / 收起来源区
3. 展开来源区后，连续勾选多个来源时不会自动收起
4. 排序切换仍然即时生效，并回到第一页
5. 搜索提交、清空仍然生效，并回到第一页
6. 移动端不会退回成三张独立控制块
7. sticky 状态与当前内容页滚动行为不冲突

## 11. 最终结论

这轮不是给内容页继续“加一个控件”，而是把现有三套控制收成一个真正紧凑的浏览控制卡。

唯一推荐实现是：

- 一张统一控制卡
- 常显来源摘要、排序和搜索
- 来源筛选按需在同卡内展开

这样可以在不改变现有功能语义的前提下，显著压缩顶部控制区高度，并让内容页更像一套经过整理的内容浏览工作台。
