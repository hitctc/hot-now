# 2026-04-08 HotNow 来源新增交互简化设计

## 1. 背景与目标

当前 `/settings/sources` 已经支持维护自定义 RSS 来源和微信公众号桥接来源，但交互层仍然暴露了太多内部字段：

- `kind`
- `来源名称`
- `来源主页`
- `bridge input mode`
- `feed URL`

这些字段对当前产品目标来说并不是用户真正关心的输入。用户要表达的真实意图只有两类：

- 我想加一个 RSS 来源
- 我想加一个公众号来源

因此本设计的目标是：

- 把 `/settings/sources` 的新增/编辑交互收口成面向意图的极简表单
- 把 `kind / 来源名称 / 来源主页 / bridge 细节` 全部下沉到系统内部自动处理
- 继续保留当前“保存时解析 bridge，采集时只认最终 `rss_url`”的架构
- 避免新增异步状态机、向导式流程或需要用户二次回填内部字段的失败交互

本设计是对 [2026-04-08-微信公众号桥接来源设计文档.md](/Users/tc-nihao/100-tc/700-code/100-center/hot-now/docs/superpowers/specs/2026-04-08-微信公众号桥接来源设计文档.md) 的交互层修订，不推翻此前“仓库内置 bridge 模块 + RSS-first 主链路”的总体结论。

## 2. 设计结论

采用下面的输入模型：

- `新增 RSS`
  - 只填 `RSS URL`
- `新增公众号`
  - 必填 `公众号名称`
  - 可选 `公众号文章链接`

其余所有字段均由系统内部生成或推断：

- `kind`
- `name`
- `site_url`
- `source_type`
- `bridge_kind`
- `bridge_config_json`
- 最终落库的 `rss_url`

保存行为保持同步：

- 保存时即做 RSS/bridge 解析
- 成功才落库
- 失败直接返回中文错误
- 不引入“待解析来源”或后台补解析状态

## 3. 方案对比与选择

### 3.1 方案 A：保留现有字段，折叠到高级设置

优点：

- 改动最小

缺点：

- 用户仍然需要理解内部字段
- 只是视觉隐藏，不是产品语义收口

### 3.2 方案 B：极简意图输入，系统自动补全

优点：

- 最符合当前使用方式
- 把系统内部概念留在系统内部
- 与当前“单人高频操作”场景更匹配

缺点：

- 后端需要补一层自动推断逻辑

### 3.3 方案 C：向导式导入

优点：

- 对完全不熟悉来源结构的用户最友好

缺点：

- 交互过重
- 不符合当前产品节奏

最终采用 `方案 B`。

## 4. 前端交互设计

### 4.1 新增来源入口

`/settings/sources` 顶部保留一个统一入口，但弹窗内部先提供两类动作：

- `新增 RSS`
- `新增公众号`

不再让用户先面对完整通用 source 表单。

### 4.2 RSS 表单

RSS 表单只保留一个输入项：

- `RSS URL`

用户不需要填写：

- 来源名称
- 来源主页
- kind

### 4.3 公众号表单

公众号表单只保留两个输入项：

- `公众号名称`（必填）
- `公众号文章链接`（可选，建议填写）

交互原则：

- 只要用户已经同时填了名称和文章链接，就视为一次性完整提交
- 不再要求用户理解 `feed URL / bridge id / input mode`
- 如果只填名称，系统会按名称检索 bridge/索引来源并尝试生成 feed

### 4.4 编辑来源

编辑自定义来源时，表单仍遵循同样的简化原则：

- RSS 来源只允许改 `RSS URL`
- 公众号来源只允许改 `公众号名称` 与 `公众号文章链接`

不再允许编辑：

- `kind`
- `来源主页`
- `bridge 细节`

### 4.5 列表展示

来源库存表与统计表继续统一展示，但只暴露必要信息：

- 来源名
- 来源类型标签（`RSS` / `公众号桥接`）
- 启用状态
- 最近抓取状态
- 最终 RSS 链接
- 编辑 / 删除操作

不再把 `bridge input mode`、内部摘要等细节作为主展示信息。

## 5. 后端处理设计

### 5.1 新的输入契约

`POST /actions/sources/create`

- RSS:
  - `sourceType = rss`
  - `rssUrl`
- 公众号:
  - `sourceType = wechat_bridge`
  - `wechatName`
  - `articleUrl?`

`POST /actions/sources/update`

- 与 create 保持相同语义
- 额外带目标 `kind`

删除、启停、展示模式接口保持不变。

### 5.2 自动推断职责

后端在真正调用 `saveSource(...)` 之前，新增一层用户输入解析：

- `resolveRssSourceInput(...)`
- `resolveWechatSourceInput(...)`

它们负责把极简表单转成标准化 source 输入。

### 5.3 RSS 自动补全规则

当用户只提交 `rssUrl` 时，系统内部自动：

- 拉取 feed 元信息
- 优先使用 feed 标题作为 `name`
- 从 feed `link` 或 URL 根域生成 `site_url`
- 从 feed 标题或 URL 生成稳定 `kind`
- 固定 `source_type = rss`

如果 feed 无法识别或不合法，则保存失败，不落库。

### 5.4 公众号自动补全规则

当用户提交公众号来源时：

- 如果有 `articleUrl`
  - 优先走文章链接解析路径
  - 调用 bridge 服务生成最终 feed
- 如果没有 `articleUrl`
  - 走名称检索路径
  - 从 bridge / 索引来源中找到候选并生成最终 feed

系统内部自动生成：

- `kind`
- `name`
- `site_url`
- 最终 `rss_url`
- `bridge_config_json`

其中：

- `name` 以用户填写的 `公众号名称` 为主
- `kind` 基于规范化名称生成稳定 slug；若冲突，则追加稳定后缀
- `site_url` 优先从文章链接或解析结果反推 `mp.weixin.qq.com` 主页语义；如果无法精确反推，允许回退到 `https://mp.weixin.qq.com/`

## 6. 错误处理与提示口径

这轮不引入二次补录流程，也不把用户拉回内部字段世界。

失败时只返回产品语言：

- `这个 RSS 地址暂时无法识别，请检查链接是否正确`
- `没有找到这个公众号的可用来源，请换一个名称或补一篇文章链接`
- `这篇文章暂时无法生成订阅源，请换一篇文章再试`
- `公众号桥接服务当前不可用，请稍后再试`

这里允许提示“补一篇文章链接”，但它是一次保存失败后的中文建议，不是把页面切回内部字段编辑模式。

## 7. 范围边界

本轮要做：

- `/settings/sources` 新增/编辑表单简化
- create/update 接口契约简化
- RSS 自动元数据补全
- 公众号名称 + 可选文章链接的同步保存解析

本轮不做：

- 公众号候选搜索列表
- 多 provider bridge 策略
- 批量导入
- 异步后台解析状态机
- 历史自定义来源自动迁移修复

## 8. 测试与验证

至少补齐：

- client
  - RSS 表单只提交 `rssUrl`
  - 公众号表单只提交 `wechatName + articleUrl?`
  - 编辑时不再出现内部字段输入
- server
  - create/update 的新 payload 校验
  - RSS 自动补全失败路径
  - 公众号名称解析路径
  - 公众号文章链接解析路径
- build
  - `npm run build:client`
  - `npm run build`

手动 smoke 重点：

- `/settings/sources` 新增一个 RSS
- `/settings/sources` 新增一个公众号（只填名称）
- `/settings/sources` 新增一个公众号（名称 + 文章链接）
- 确认列表里出现的是系统自动生成后的标准 source，而不是要求用户补内部字段

## 9. 风险与取舍

这轮最大的取舍是：

- 为了换取更简单的表单，后端需要承担更多推断责任

当前这个取舍是值得的，因为用户关心的是“把来源接进来”，不是“手工维护 source 主键和 bridge 配置”。

名称检索路径的稳定性会低于文章链接解析路径，因此：

- `公众号名称` 仍然允许单独提交
- 但页面可以继续弱提示“文章链接可提高成功率”

这不改变“一次性提交”的交互原则，只是帮助用户提高首次成功率。
