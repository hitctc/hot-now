# 2026-04-23 B 站搜索与微博热搜设计文档

## 1. 背景与目标

在已经完成 `Twitter 账号采集`、`Twitter 关键词搜索`、`Hacker News 搜索` 之后，下一阶段继续补齐国内热度侧信号。

本阶段覆盖两条来源链路：

1. `B 站搜索`
2. `微博热搜榜匹配（固定 AI 关键词，只进入 AI 热点，不做微博全文搜索）`

这一阶段的目标不是把更多来源简单堆进系统，而是继续沿用当前已经验证过的模式：

- 非 RSS 来源继续走独立 collector 通道
- 配置和普通 `content_sources` 分离
- 采集失败只影响单一来源，不阻断其他来源
- 后台控制保持克制，不把 `/settings/sources` 做成越来越重的配置平台

## 2. 本阶段结论

本阶段采用以下收口方案：

### 2.1 B 站搜索

- 设计为独立来源类型：`bilibili_search`
- 后台维护独立 query 列表
- 只支持手动采集
- 第一版只搜索 `视频结果`
- 采集结果允许进入 `AI 新讯` 和 `AI 热点`

### 2.2 微博热搜榜匹配（固定 AI 关键词，只进入 AI 热点，不做微博全文搜索）

- 设计为独立来源类型：`weibo_trending`
- 不开放后台关键词配置
- 只抓取微博热搜榜
- 用内置固定 AI 关键词集合做榜单匹配
- 命中结果只进入 `AI 热点`
- 不进入 `AI 新讯`
- 不做微博全文搜索

### 2.3 实现顺序

虽然这两个来源在同一阶段统一设计，但实现顺序明确为：

1. 先实现 `B 站搜索`
2. 再实现 `微博热搜榜匹配`

原因很直接：B 站更接近已经落地的 `HN query 搜索` 模式，结构风险更低，便于复用当前 collector 和后台工作台设计。

## 3. 不采用的方案

### 3.1 不把 B 站和微博都做成统一搜索配置平台

不采用原因：

- B 站是 query 搜索
- 微博这条本质是热榜匹配
- 两者的语义不同，强行统一只会让后台语义变差

### 3.2 不给微博开放后台关键词 CRUD

不采用原因：

- 微博这一阶段只是补充热度信号，不是主搜索通道
- 如果也开放可维护关键词，`/settings/sources` 会继续变重
- 第一版优先验证“微博热搜对 AI 热点是否真的有价值”，不是优先追求灵活性

### 3.3 不把微博当全文搜索做

不采用原因：

- 这会把热搜榜匹配和微博搜索混成两种不同产品
- 调研已经确认微博本阶段定位是“热点信号补充”，不是微博正文发现工具
- 一旦走微博全文搜索，复杂度、噪音和维护成本都会明显上升

## 4. 数据模型设计

本阶段新增一张独立配置表：

### 4.1 `bilibili_queries`

字段建议：

- `id`
- `query`
- `priority`
- `is_enabled`
- `notes`
- `last_fetched_at`
- `last_success_at`
- `last_result`
- `created_at`
- `updated_at`

约束：

- `query` 唯一，大小写不敏感
- `priority` 继续沿用 `0-100`
- 排序继续按 `priority DESC, query ASC`

说明：

- 这张表只服务 B 站搜索，不混入 `content_sources`
- 第一版不做“分类”“展示启用”“单 query 时间窗口”等附加字段

### 4.2 微博不新增配置表

微博这一阶段不新建 `weibo_keywords` 之类的配置表。

固定 AI 关键词集合直接放在代码常量里，由 collector 使用。这样可以把复杂度控制在最低范围内。

### 4.3 隐藏聚合 source

为满足 `content_items.source_id` 外键，继续维护隐藏聚合 source。

本阶段新增两个聚合 source：

- `bilibili_search`
- `weibo_trending`

约束：

- 都不承载实际配置
- 都不出现在普通 source 库存表中
- `bilibili_search` 的内容可在内容页作为来源筛选项暴露
- `weibo_trending` 的内容第一版不要求额外二级筛选

## 5. B 站搜索设计

### 5.1 来源定位

`B 站` 在本项目里的定位是：

- 国内视频内容信号补充源
- 不是 RSS
- 不是账号订阅
- 不是 UP 主管理平台
- 第一版仅作为 query 搜索 collector

### 5.2 配置方式

`/settings/sources` 增加独立 `B 站搜索` 分区。

第一版支持：

- 新增 query
- 编辑 query
- 删除 query
- 启停 query
- 手动采集
- 查看最近成功
- 查看最近结果

第一版不支持：

- 分类
- 展示启用单独开关
- 单 query 限额配置
- 内容页二级 query 筛选
- 自动调度

### 5.3 搜索边界

第一版只搜 `视频结果`，不搜：

- UP 主
- 专栏
- 番剧
- 直播

搜索 API 采用公开视频搜索接口，按最新内容优先获取结果。

### 5.4 全局采集参数

第一版固定全局参数，不开放后台配置：

- 每轮最多处理 `5` 个已启用 query
- 每个 query 最多取前 `10` 条结果
- 默认按最新结果优先

### 5.5 入库映射

B 站结果统一映射到现有 `content_items`。

核心映射字段：

- `title`
- `url`
- `author / uploader`
- `publishedAt`
- `summary`
- `externalId`

`metadata_json` 里保留：

- `collector.kind = "bilibili_search"`
- `queryId`
- `query`
- `priority`
- `bvid`
- `mid`
- `play`
- `danmaku`
- `comment`
- `like`
- `favorite`
- `matchedQueries`

### 5.6 页面去向

B 站内容允许进入：

- `AI 新讯`
- `AI 热点`

是否最终可见，仍由现有内容筛选与热点逻辑决定，不单独为 B 站加一套特殊规则。

## 6. 微博热搜榜匹配（固定 AI 关键词，只进入 AI 热点，不做微博全文搜索）设计

### 6.1 来源定位

微博在本项目里的定位是：

- 大众传播层热度信号
- 热搜榜匹配来源
- 不是微博全文搜索
- 不是正文采集器
- 不是 AI 新讯主阅读流

### 6.2 配置方式

第一版微博不开放后台关键词维护。

后台只需要保留：

- 一个独立 `微博热搜榜匹配` 分区
- 一个手动采集按钮
- 最近成功
- 最近结果

如果需要，可以在分区说明里明确写出：

- 当前使用固定 AI 关键词集合
- 本阶段只进入 `AI 热点`
- 不做微博全文搜索

### 6.3 数据来源

微博使用公开热搜榜接口，拉取榜单后做本地关键词匹配。

固定 AI 关键词集合由代码维护，第一版覆盖 AI 主线即可，例如：

- `AI`
- `人工智能`
- `大模型`
- `OpenAI`
- `ChatGPT`
- `Anthropic`
- `Claude`
- `Gemini`
- `机器人`
- `智能体`

具体集合以实现阶段代码常量为准，但边界不变：

- 只允许固定集合
- 不开放后台维护

### 6.4 匹配语义

微博 collector 的输出不是“搜索结果”，而是“热搜榜命中结果”。

因此它的结果语义应明确为：

- 当前热搜榜上出现了哪些 AI 相关话题
- 每个话题的热度是多少
- 可回链到微博搜索或热搜详情页

### 6.5 入库映射

微博结果也统一映射到现有 `content_items`。

核心映射字段：

- `title`
- `url`
- `summary`
- `publishedAt` 或 `fetchedAt`
- `externalId`

`metadata_json` 里保留：

- `collector.kind = "weibo_trending"`
- `topic`
- `hotValue`
- `matchedKeyword`
- `rawRank`

### 6.6 页面去向

微博结果只进入：

- `AI 热点`

不进入：

- `AI 新讯`

原因：

- 它更像热点信号
- 不适合作为主阅读流内容源

## 7. 后端链路设计

### 7.1 Collector 类型扩展

本阶段新增两类 collector：

- `bilibili_search`
- `weibo_trending`

继续沿用当前非 RSS collector 模式，不把它们伪装成 RSS。

### 7.2 执行入口

第一版都只提供独立手动入口：

- `POST /actions/bilibili/collect`
- `POST /actions/weibo/collect`

它们都不并入默认 `/actions/collect`。

### 7.3 失败隔离

两条来源都必须满足：

- 失败只影响本来源
- 不影响 RSS / Twitter / HN
- 最近结果能回写到后台
- 后端返回明确 reason，前端能翻译成可读提示

### 7.4 命中关系

B 站 query 搜索会像 HN 一样存在“同一条内容命中多个 query”的情况。

因此第一版继续复用当前 `matchedQueries` 合并思路：

- 新内容入库时写入命中 query
- 已存在内容再次命中时合并 query 列表

微博这条第一版不需要做复杂命中关系表，只保留 `matchedKeyword` 到 `metadata_json` 即可。

## 8. 前端工作台设计

`/settings/sources` 在本阶段新增两个独立分区：

1. `B 站搜索`
2. `微博热搜榜匹配（固定 AI 关键词，只进入 AI 热点，不做微博全文搜索）`

### 8.1 B 站分区

展示：

- query 总数
- 采集启用数
- API 状态说明
- 手动采集按钮
- query 表格

表格列：

- 查询词
- 优先级
- 采集启用
- 最近成功
- 最近结果
- 操作

### 8.2 微博分区

展示：

- 固定关键词策略说明
- 只进入 AI 热点说明
- 手动采集按钮
- 最近成功
- 最近结果

微博第一版可以不做列表表格，因为没有后台可维护项。

## 9. 测试与验收

### 9.1 B 站

至少覆盖：

- migration 与 repository
- API client 请求参数和响应归一化
- collector 搜索与失败隔离
- 手动采集入库
- `/settings/sources` 的 CRUD 与手动采集 action
- 页面分区渲染与基本交互

### 9.2 微博

至少覆盖：

- API client 响应归一化
- 固定关键词匹配逻辑
- 只进 `AI 热点` 的链路约束
- 手动采集 action
- `/settings/sources` 微博分区读模型和按钮交互

### 9.3 验收标准

这一阶段完成后，应满足：

1. `B 站` 已支持独立 query 配置、手动采集和内容入库。
2. `B 站` 第一版只搜视频结果。
3. `B 站` 内容可进入 `AI 新讯` 和 `AI 热点`。
4. `微博` 已支持热搜榜手动采集。
5. `微博` 只使用固定 AI 关键词集合匹配榜单。
6. `微博` 只进入 `AI 热点`。
7. `微博` 明确不做微博全文搜索。
8. 两条来源的失败都不会阻断其他来源。

## 10. 本阶段明确不做

本阶段不做以下内容：

- B 站账号采集
- B 站 UP 主维度管理
- B 站专栏 / 番剧 / 直播搜索
- B 站自动调度
- B 站内容页二级 query 筛选
- 微博全文搜索
- 微博后台关键词 CRUD
- 微博自动调度
- 微博进入 `AI 新讯`
- B 站和微博的正文抓取
- 搜狗 HTML 爬虫

## 11. 推荐实现顺序

1. 先完成 `B 站搜索` 的 spec 对应实现
2. 再完成 `微博热搜榜匹配（固定 AI 关键词，只进入 AI 热点，不做微博全文搜索）`

原因：

- B 站可直接复用已存在的 `query 搜索` 结构
- 微博语义不同，更适合在 B 站之后单独落地
- 这样可以避免把两个来源同时塞进同一轮实现，造成 scope 漂移
