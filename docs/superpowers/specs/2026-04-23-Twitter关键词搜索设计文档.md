# 2026-04-23 HotNow Twitter 关键词搜索设计

## 1. 背景与目标

当前 HotNow 已经完成两类主路径：

- `RSS / 公众号 bridge -> content_sources -> 统一采集`
- `Twitter 账号列表 -> twitter_accounts -> 手动账号采集`

Twitter 账号采集适合盯固定账号，但它覆盖不了下面这类场景：

- 某个 AI 热词突然升温，但不在已配置账号内
- 同一事件由媒体、从业者和匿名账号同时扩散，需要补漏
- 想先用少量关键词手动验证搜索价值，再决定是否扩大范围

因此下一步需要新增一条独立于账号采集的 Twitter 关键词搜索链路。

本轮目标是：

- 在 `/settings/sources` 中新增一个独立的 `Twitter 关键词搜索` 分区
- 支持维护关键词列表，并以中文口径展示配置状态
- 只支持手动触发采集，不参与默认 `10` 分钟定时采集
- 对单次采集做明确的 credits 限流，避免异常消耗
- 采集结果继续写入现有 `content_items`
- 允许通过“关键词级展示开关”控制这类内容是否进入 `AI 新讯 / AI 热点`

本轮明确不做：

- 自动定时采集
- 语言 / 地区配置
- 关键词级时间窗配置
- 单条内容级隐藏
- LLM 过滤接入
- Twitter 实时流或 webhook

## 2. 设计结论

采用一条与 Twitter 账号采集平行、但配置和展示语义独立的链路：

1. 新增 `twitter_search_keywords` 保存关键词配置
2. 新增 `twitter_search_keyword_matches` 记录 tweet 与关键词的命中关系
3. 保留一个隐藏聚合 source `twitter_keyword_search`，只用于满足 `content_items.source_id` 外键
4. 新增独立手动动作 `POST /actions/twitter-keywords/collect`
5. 单次执行只处理最多 `5` 个已启用关键词，每个关键词最多写入前 `10` 条搜索结果
6. 所有 tweet 继续按 `external_id = twitter:{tweetId}` 全局去重，避免与 Twitter 账号采集重复入库
7. 内容页展示时，只有至少命中一个“展示启用”的关键词，这条内容才允许通过关键词搜索路径显示

这意味着 Twitter 关键词搜索不会被塞进现有 `content_sources` 普通来源工作台，也不会复用 RSS 来源的启停语义。它是一套独立的“配置对象 + 手动采集 + 命中关联”模型。

## 3. 用户已确认的边界

本设计基于下面这些已确认约束：

- 关键词搜索必须是 `Twitter 账号` 之外的独立分区
- 第一版关键词配置只保留最小字段：
  - `关键词`
  - `分类`
  - `优先级`
  - `采集启用`
  - `展示启用`
  - `备注`
- 页面中的表格列、下拉选项和状态文案统一使用中文描述
- 第一版只支持手动采集
- 展示控制是关键词级，不是单条内容级
- `采集启用` 和 `展示启用` 必须拆成两个开关
- 单次采集必须做双限流：
  - 最多处理 `5` 个关键词
  - 每个关键词最多处理 `10` 条结果

这些点都属于本轮硬边界，不再继续开放成可配置项。

## 4. 数据模型

### 4.1 twitter_search_keywords

新增表 `twitter_search_keywords`，字段建议如下：

- `id`
- `keyword`
- `category`
- `priority`
- `is_collect_enabled`
- `is_visible`
- `notes`
- `last_fetched_at`
- `last_success_at`
- `last_result`
- `created_at`
- `updated_at`

字段语义：

- `keyword`
  - 原始搜索词
  - 允许用户直接输入简短词组或带空格短语
- `category`
  - 仍使用现有 Twitter 账号分类枚举体系，避免平行定义
- `priority`
  - 影响单次手动采集的执行顺序
- `is_collect_enabled`
  - 关闭后不再参与后续手动采集
- `is_visible`
  - 关闭后，该关键词命中的内容不再通过关键词搜索路径进入内容页
- `last_result`
  - 保存最近一次采集摘要，例如“本次抓取成功，获得 4 条可入库推文”

建议约束：

- `keyword` 做 `trim` 后不能为空
- `keyword` 建唯一索引，避免同词重复维护
- `priority` 默认 `60`

### 4.2 twitter_search_keyword_matches

新增表 `twitter_search_keyword_matches`，字段建议如下：

- `id`
- `keyword_id`
- `tweet_external_id`
- `content_item_id`
- `created_at`
- `updated_at`

建议约束：

- 唯一键：`(keyword_id, tweet_external_id)`
- `keyword_id` 外键指向 `twitter_search_keywords.id`
- `content_item_id` 外键指向 `content_items.id`

这张表负责解决两个问题：

1. 同一条 tweet 可以被多个关键词命中
2. 内容页展示需要根据“至少命中一个展示启用关键词”来判断是否可见

如果没有这张表，关键词级展示开关就无法稳定成立。

### 4.3 隐藏聚合 source

继续沿用 Twitter 账号采集的做法，在 `content_sources` 中保留一个隐藏聚合来源：

- `kind = twitter_keyword_search`

作用只有两个：

- 给 `content_items.source_id` 提供合法外键
- 让内容 API 能识别这是一类 Twitter 搜索内容

它不进入普通 RSS 来源管理列表，也不允许作为普通 source 被用户编辑。

## 5. 分类与中文映射

底层仍存英文枚举，页面统一显示中文：

| 存储值 | 页面显示 |
|---|---|
| `official_vendor` | 官方厂商 |
| `product` | 产品 |
| `person` | 关键人物 |
| `topic` | 主题 |
| `media` | 媒体 |
| `other` | 其他 |

这一层映射要与 Twitter 账号分区保持一致，不再新造第二套中文体系。

## 6. 采集流程

### 6.1 触发方式

第一版只提供手动触发：

- `POST /actions/twitter-keywords/collect`

它不应被下面这些入口触发：

- 默认 `10` 分钟采集调度
- `POST /actions/collect`
- `POST /actions/run`

也就是说，用户必须显式进入 `Twitter 关键词搜索` 分区后点击“手动采集 Twitter 关键词”，才会消耗 TwitterAPI.io credits。

### 6.2 关键词选择规则

一次手动采集的关键词选择规则固定为：

1. 只读取 `is_collect_enabled = true`
2. 按 `priority DESC, id ASC` 排序
3. 最多只取前 `5` 个关键词

这样做的原因很直接：

- 用户可以通过优先级决定“本轮先搜谁”
- 关键词数量上限稳定，不会因后台积累过多配置而失控

### 6.3 单关键词搜索规则

每个关键词执行一次 TwitterAPI.io `advanced_search` 请求，默认策略固定为：

- `queryType = Latest`
- 不做分页
- 只搜索最近固定时间窗内的内容
- API 返回后本地截断到前 `10` 条

时间窗不作为第一版配置项，统一由服务端固定，例如最近 `72` 小时。这样可以避免热词拉出过深历史结果，也能保证成本和新鲜度稳定。

### 6.4 credits 限流

本轮固定的双限流规则：

- 单次手动采集最多 `5` 个关键词
- 每个关键词最多处理 `10` 条结果

系统需要在手动采集完成后返回摘要，例如：

- 已启用关键词：`8`
- 本次执行关键词：`5`
- 本次搜索返回 tweet：`23`
- 本次新增入库：`7`
- 复用已有内容：`12`
- 失败关键词：`1`

这类摘要必须直接回显在页面上，避免用户只看到“成功”而不知道 credits 实际消耗和数据价值。

## 7. 去重与入库策略

### 7.1 全局 tweet 去重

Twitter 账号采集和 Twitter 关键词搜索必须共用同一条 tweet 唯一键：

- `external_id = twitter:{tweetId}`

处理规则：

- 如果 tweet 不存在：
  - 新建 `content_items`
  - `source_id` 指向隐藏聚合 source `twitter_keyword_search`
- 如果 tweet 已存在：
  - 不新建第二条 `content_items`
  - 只补 `twitter_search_keyword_matches`

这样可以避免：

- 同一条 tweet 在账号采集和关键词搜索之间重复入库
- 内容页出现重复卡片
- 后续对同一条内容做反馈时分裂成两条记录

### 7.2 与 Twitter 账号采集的关系

如果同一条 tweet 先被 `Twitter 账号采集` 抓到，再被 `关键词搜索` 命中：

- 保留原有 `content_items`
- 不覆盖已有 `source_id`
- 只新增关键词命中关系

如果同一条 tweet 先由 `关键词搜索` 入库，后面又被 `账号采集` 抓到：

- 不新增第二条内容
- 允许账号采集侧补它自己的来源关系或状态字段

换句话说，关键词搜索是“补漏入口”，不是新的内容主键体系。

### 7.3 内容字段映射

Twitter 搜索结果写入 `content_items` 时，映射继续尽量复用已有 Twitter 账号采集逻辑：

- `title`
  - 使用 tweet 文本的摘要标题
- `canonical_url`
  - 指向 tweet 永久链接
- `external_id`
  - `twitter:{tweetId}`
- `published_at`
  - tweet 发布时间
- `summary`
  - tweet 全文或规整后的摘要

如果 Twitter 账号采集已经有一套共享的 tweet 规整函数，本轮优先复用，不再复制第二份格式化逻辑。

## 8. 展示语义

### 8.1 关键词级双开关

本轮只支持关键词级控制：

- `采集启用`
  - 决定后续还搜不搜这个关键词
- `展示启用`
  - 决定这个关键词命中的内容是否还能进入内容页

两者必须独立：

- 可以停采但继续展示历史内容
- 也可以保留采集，但临时隐藏这组内容

### 8.2 内容页可见性规则

对一条由关键词搜索命中的内容，判断是否可见时执行下面的逻辑：

1. 查询它关联的所有 `twitter_search_keyword_matches`
2. 只要存在至少一个 `is_visible = true` 的关键词
3. 这条内容就允许通过关键词搜索路径进入 `AI 新讯 / AI 热点`

反过来，如果一条内容只命中了已隐藏关键词，则它不再通过关键词搜索路径显示。

### 8.3 与账号采集内容的叠加规则

如果一条 tweet 同时属于：

- 已配置 Twitter 账号采集命中
- 关键词搜索命中

那么关键词的 `展示启用` 不应把它从账号采集路径一并隐藏。否则用户会得到“明明监控了账号，却因为关键词隐藏导致内容消失”的错误结果。

因此本轮的正确语义是：

- `关键词展示开关` 只影响“通过关键词搜索路径补进来的可见性”
- 不反向干扰 Twitter 账号采集自身的展示资格

## 9. 后端接口设计

建议新增接口：

- `POST /actions/twitter-keywords/create`
- `POST /actions/twitter-keywords/update`
- `POST /actions/twitter-keywords/delete`
- `POST /actions/twitter-keywords/toggle-collect`
- `POST /actions/twitter-keywords/toggle-visible`
- `POST /actions/twitter-keywords/collect`

`/api/settings/sources` 读模型需要补充：

- `twitterSearchKeywords`
- `operations.canTriggerManualTwitterKeywordCollect`

接口层要求：

- 返回中文化的结果摘要，便于前端直接渲染
- 删除属于危险操作，延续现有高风险动作二次确认约束
- `collect` 接口返回本次运行统计，不要只返回布尔成功

## 10. /settings/sources 页面设计

在现有 `Twitter 账号` 分区后新增一个独立分区：

- 标题：`Twitter 关键词搜索`

分区顶部展示四块概览：

- 关键词总数
- 采集启用数
- 展示启用数
- API 状态

操作按钮：

- `新增关键词`
- `手动采集 Twitter 关键词`

表格列：

- `关键词`
- `分类`
- `优先级`
- `采集启用`
- `展示启用`
- `最近成功`
- `最近结果`
- `操作`

第一版不做分页、批量导入或分组视图。关键词数量不大时，简单表格更利于直接管理。

## 11. 错误处理与状态反馈

关键词搜索常见异常必须明确区分：

- `TWITTER_API_KEY` 未配置
- TwitterAPI.io 返回 `401` / `403`
- credits 不足或被限频
- 搜索成功但没有可入库的新内容
- 搜索成功，但结果全部被去重

`last_result` 与手动采集摘要要能覆盖这几类信息，避免再出现“最近成功已更新，但用户看不到任何数据，也不知道是 0 条还是被过滤掉”的误导。

建议状态文案风格统一为：

- `本次抓取成功，获得 3 条可入库推文。`
- `本次抓取成功，但没有可入库的新推文。`
- `本次抓取失败：TwitterAPI.io request failed with 429`

## 12. 测试要求

本轮实现时至少补下面几类测试：

1. 数据层测试
   - 关键词表增删改查
   - `keyword` 唯一约束
   - 关键词命中关系去重

2. 采集服务测试
   - 只选前 `5` 个已启用关键词
   - 每词只处理前 `10` 条结果
   - 同一 tweet 与账号采集 / 关键词搜索之间不重复入库
   - 搜索成功但无新增时返回正确摘要

3. 内容页读模型测试
   - `is_visible = false` 的关键词不会让仅靠该关键词命中的内容继续显示
   - 同一 tweet 命中多个关键词时，只要还有一个可见关键词就继续显示
   - 账号采集内容不会被关键词隐藏规则误伤

4. 系统页前后端测试
   - `/api/settings/sources` 正确返回关键词列表和操作权限
   - `/settings/sources` 正确渲染新分区、中文分类和双开关状态
   - 手动采集按钮能显示中文摘要

## 13. 实施顺序建议

进入实现阶段后，建议按下面顺序落地：

1. 先补 migration 和 repository
2. 再做 Twitter 关键词采集服务
3. 再接 `/api/settings/sources` 读模型和 actions
4. 最后接 `/settings/sources` 页面与交互
5. 结束前补内容页可见性测试和手动 smoke test

这能把风险控制在最小范围内，也便于在每一步检查 credits 消耗与入库行为是否符合预期。

## 14. 本轮不扩展的方向

下面这些方向已经明确排除，不要在实现时顺手长出来：

- 关键词级地区 / 语言参数
- 关键词级时间窗自定义
- 自动调度
- 多页翻页抓取
- 单条内容级隐藏
- 关键词组、标签或批量管理
- 把关键词搜索并回普通 `content_sources`
- 接入 LLM 做二次过滤

如果后续需要这些能力，应在本设计基础上再开下一份 spec，而不是在本轮里顺手扩容。
