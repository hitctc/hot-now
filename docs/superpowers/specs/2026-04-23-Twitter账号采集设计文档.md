# Twitter 账号采集设计文档

日期：2026-04-23

## 背景

当前 hot-now 已有 RSS 和微信公众号桥接来源，来源配置集中在 `/settings/sources` 和 `content_sources`。下一阶段要引入多源主动采集，其中第一阶段只做 Twitter 账号采集，用于优先获取 AI 厂商、AI 产品和关键人物的一手动态。

经过讨论，本阶段不把 Twitter 账号硬塞进 `content_sources`。原因是后续还会接入 Twitter 关键词、Hacker News、B 站和微博热搜，如果所有类型都共用一张来源配置表，后台会越来越难管理。

## 目标

本阶段交付一个可独立管理的 Twitter 账号采集能力：

- 后台页面可配置 Twitter 账号列表。
- 支持新增、编辑、启用、停用账号。
- 支持账号分类、优先级和备注。
- 采集结果统一进入现有 `content_items`，不新建另一套内容展示链路。
- `TWITTER_API_KEY` 缺失或接口失败时，只影响 Twitter 账号采集，不影响 RSS 和微信公众号采集。

## 不做范围

本阶段不做：

- Twitter 关键词搜索。
- Hacker News、B 站、微博热搜。
- 搜狗 HTML 爬虫。
- Twitter webhook / stream 实时监听。
- 在后台页面保存 TwitterAPI.io API key。
- 自动内置一大批不可编辑账号。

## 外部接口

第一版使用 TwitterAPI.io 用户最新推文接口：

- Endpoint：`GET https://api.twitterapi.io/twitter/user/last_tweets`
- Header：`X-API-Key: <api-key>`
- Query：
  - `userId`：用户 ID，官方文档说明比 `userName` 更稳定、更快。
  - `userName`：账号 screen name。
  - `cursor`：分页 cursor，首页为空字符串。
  - `includeReplies`：是否包含回复，默认 `false`。
- Response：
  - `tweets[]`
  - `has_next_page`
  - `next_cursor`
  - `status`
  - `message`

设计取舍：

- 后台先要求用户填写 `username`，系统可在采集成功后缓存返回的 `author.id`。
- 每个账号每轮先拉第一页，最多 20 条。后续如果账号数超过 20 且需要接近实时，再单独评估 stream 方案。
- 默认 `includeReplies=false`，避免回复内容污染 AI 新讯。

参考：<https://docs.twitterapi.io/api-reference/endpoint/get_user_last_tweets>

## 数据模型

新增独立表 `twitter_accounts`，专门保存 Twitter 账号采集配置。

建议字段：

```sql
CREATE TABLE twitter_accounts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT NOT NULL UNIQUE,
  user_id TEXT,
  display_name TEXT NOT NULL,
  category TEXT NOT NULL,
  priority INTEGER NOT NULL DEFAULT 50,
  include_replies INTEGER NOT NULL DEFAULT 0,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  notes TEXT,
  last_fetched_at TEXT,
  last_success_at TEXT,
  last_error TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

字段说明：

- `username`：用户在 X/Twitter 上的 screen name，不带 `@`。
- `user_id`：接口返回的稳定用户 ID，采集成功后可回填。
- `display_name`：后台展示名称，默认等于 `username`。
- `category`：账号分类，建议第一版支持 `official_vendor`、`product`、`person`、`media`、`other`。
- `priority`：账号优先级，数值越高越重要。
- `include_replies`：默认不采集回复。
- `is_enabled`：是否参与采集。
- `last_fetched_at`、`last_success_at`、`last_error`：用于后台解释当前状态。

采集结果仍写入现有 `content_items`。Twitter 平台指标先通过新增 `metadata_json` 承载，避免把内容表拆成平台专用结构。

建议给 `content_items` 新增：

```sql
ALTER TABLE content_items ADD COLUMN metadata_json TEXT;
```

Twitter 内容的 `metadata_json` 示例：

```json
{
  "collector": {
    "kind": "twitter_account",
    "accountId": 1,
    "username": "sama"
  },
  "metrics": {
    "retweetCount": 12,
    "replyCount": 8,
    "likeCount": 200,
    "quoteCount": 3,
    "viewCount": 10000,
    "bookmarkCount": 15
  },
  "author": {
    "id": "123",
    "username": "sama",
    "name": "Sam Altman",
    "followers": 3000000,
    "verified": true
  }
}
```

## 后台页面

继续使用 `/settings/sources` 作为采集管理入口，但页面内部拆成分区，而不是把所有来源混成一个表。

第一阶段页面结构：

- RSS / 微信公众号来源：沿用现有列表和表单。
- Twitter 账号：新增独立分区。

Twitter 账号分区能力：

- 列表字段：
  - 展示名称
  - username
  - 分类
  - 优先级
  - 启用状态
  - 最近成功时间
  - 最近错误
- 表单字段：
  - username，必填，不带 `@`
  - 展示名称，可选，默认 username
  - 分类，必填，默认 `official_vendor`
  - 优先级，默认 50
  - 是否采集回复，默认关闭
  - 备注，可选
- 操作：
  - 新增账号
  - 编辑账号
  - 启用 / 停用账号
  - 删除账号

删除账号属于配置删除，不删除已入库的历史内容。

## 采集链路

新增 Twitter 账号 collector，不改 RSS adapter 的基本职责。

建议主链路：

```text
runCollectionCycle
  -> loadEnabledSourceIssues      # 现有 RSS / 微信公众号
  -> collectTwitterAccountItems   # 新增 Twitter 账号
  -> 合并为统一候选内容
  -> 去重
  -> 新鲜度过滤
  -> 优先级排序
  -> AI 过滤
  -> upsertContentItems
  -> 聚合 / 报告 / 内容页
```

第一版可以把 Twitter 推文映射为现有 `CandidateItem`：

- `title`：推文正文截断生成标题。
- `sourceUrl`：推文 URL。
- `sourceName`：`X / {displayName}` 或 `Twitter / {displayName}`。
- `externalId`：`twitter:{tweet.id}`。
- `publishedAt`：`tweet.createdAt`。
- `summary`：推文正文。
- `metadata_json`：平台指标、作者信息、账号配置 ID。

如果推文里包含外链：

- 第一版仍以推文本身为主内容。
- 外链可以记录在 `metadata_json.entities.urls`。
- 不在第一版里额外抓取外链正文，避免采集链路一次引入太多不确定性。

## 去重与新鲜度

去重规则：

- `externalId = twitter:{tweet.id}` 是第一优先去重键。
- 同一 URL 重复出现时，不重复入库。
- 重复运行同一账号采集，不应新增重复内容。

新鲜度规则：

- 采集阶段保留最近一段窗口内的推文，第一版建议最多保留最近 7 天候选。
- 内容页 `AI 新讯` 仍按现有 24 小时窗口筛选。
- `AI 热点` 可用指标和优先级参与排序，但第一版不改变现有热点形成规则。

## 优先级

账号配置的 `priority` 是来源级权重。

建议默认：

- `official_vendor`：90
- `product`：80
- `person`：75
- `media`：60
- `other`：50

第一版只把 priority 写入候选或 metadata，不急着大改评分公式。后续接 AI 过滤和热点排序时，再把账号优先级、互动指标和新鲜度统一纳入评分。

## 错误处理

`TWITTER_API_KEY`：

- 只从环境变量读取。
- 不写入仓库配置文件。
- 不在日志或错误消息里输出真实值。

错误行为：

- key 缺失：Twitter 账号采集跳过，记录统一的可解释错误。
- 单个账号失败：只标记该账号 `last_error`，其他账号继续。
- 全部 Twitter 账号失败：RSS / 微信公众号采集仍继续。
- TwitterAPI.io 返回 `status=error`：保存 `message` 的安全摘要。

## API 设计

新增服务端接口建议：

- `GET /api/settings/twitter-accounts`
- `POST /actions/twitter-accounts/create`
- `POST /actions/twitter-accounts/update`
- `POST /actions/twitter-accounts/delete`
- `POST /actions/twitter-accounts/toggle`

这些接口延续当前系统页写操作的登录保护策略。

请求体第一版只接受非敏感字段，不接受 API key。

## 测试矩阵

后端测试：

- migration 会创建 `twitter_accounts` 表。
- migration 会给 `content_items` 增加 `metadata_json`。
- Twitter 账号 repository 可以新增、编辑、启停、删除账号。
- username 会被标准化，去掉前导 `@`。
- `TWITTER_API_KEY` 缺失时 collector 返回可解释 failure。
- TwitterAPI.io 成功响应会被映射为候选内容。
- 同一 tweet id 重复采集不会重复入库。
- 单个账号失败不影响其他账号。

- `/settings/sources` 显示 Twitter 账号分区。
- 新增账号表单会生成正确 payload。
- 启停账号操作会刷新列表状态。
- 最近错误能在列表里展示。

## 实施切口

建议拆成两个最小可审查步骤：

1. 先做配置闭环。
   - 新增 `twitter_accounts` 表。
   - 新增 repository 和服务端 actions。
   - `/settings/sources` 增加 Twitter 账号分区。
   - 暂不真实采集。

2. 再做采集闭环。
   - 新增 TwitterAPI.io client。
   - 新增 Twitter 账号 collector。
   - 接入 `runCollectionCycle`。
   - 推文入库并写 `metadata_json`。

这样第一步可以先确认后台管理体验，第二步再处理外部 API 和采集链路风险。

实现计划见：[Twitter 账号采集实现计划](../plans/2026-04-23-Twitter账号采集实现计划.md)。

## 决策结论

本阶段采用：

```text
独立 twitter_accounts 配置表
  -> Twitter 账号 collector
  -> 统一候选内容
  -> 统一入库 content_items
  -> 复用 AI 新讯 / AI 热点展示
```

这个方案比复用 `content_sources.bridge_config_json` 初期多一点代码，但长期更适合管理多种采集渠道，也符合当前“不同来源类型分开管理”的产品方向。
